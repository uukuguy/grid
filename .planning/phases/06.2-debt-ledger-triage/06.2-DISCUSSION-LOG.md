## Triage tag schema

**Context surfaced during discussion**:
- Existing LEDGER has 4 columns: `ID | 标题 | 处理位置 (status) | 影响 (impact)`
- Column 3 already holds emoji-status vocabulary: `🧹 tech-debt`, `🔴 phase3-gated`, `⏸️ frozen`, `🤔 revisit-after-X`
- 128 open D-rows + 52 already-CLOSED rows = 159 table rows total

| Option | Description | Selected |
|--------|-------------|----------|
| Prepend tag to column 3 status | Format: `[P2-next-milestone] 🧹 tech-debt rationale: per ADR-V2-024 双轴 dormant`. Keeps existing emoji semantic, adds machine-greppable `[P\d-...]` prefix, rationale follows. 128 atomic in-place edits. Easy regex extraction for INBOX parser. Doesn't break existing scanner habits. | ✓ |
| Add 5th column 'triage' | Format: `\| D33 \| title \| old status \| impact \| [P2] rationale`. Adds a column to the table (markdown table widening across 159 rows). More structured. But 159 row edits not 128. Bigger diff. | |
| Replace column 3 with bare tag + rationale | Format: `\| D33 \| title \| [P2-next-milestone] rationale: per ADR-V2-024 \| impact`. Loses existing emoji/state vocabulary. Cleanest but information loss. | |

**User's choice:** Prepend tag to column 3 status (Recommended)
**Notes:** Preserves emoji vocabulary; smallest diff (128 row edits vs 159); INBOX parser-friendly via regex.

## DEAD definition edge cases

**Context surfaced during discussion**:
- ROADMAP SC#1 lists 3 DEAD criteria: octo-leftover, deleted-code, ADR-superseded
- Borderline cases: D62-64 "Sandbox Tiers phase3-gated" (capability never built); D66 hermes monkey-patch (frozen V2-017 — covered by existing (c)); D108+ grid-platform/desktop/web* dormant rows
- ADR-V2-024 双轴 framework marks grid-platform / grid-desktop / web* dormant

| Option | Description | Selected |
|--------|-------------|----------|
| Add 4th criterion: 'dormant-by-ADR' | DEAD criterion (d): row is blocked on capability that ADR-V2-024 双轴 (or any Accepted ADR) marks as dormant/deferred-indefinitely. Catches D62-64 Sandbox Tiers + grid-platform/grid-desktop/web* rows. Rationale must cite the specific ADR section. Risk: 'indefinite' is fuzzy — mitigate by requiring the ADR explicitly state dormancy. | ✓ |
| Keep 3 criteria strict | Only DEAD if octo-leftover OR deleted-code OR ADR-superseded. Sandbox-Tiers / dormant-product rows stay as P3 with rationale. More conservative but bloats P3 list. | |
| Add criterion + boundary check | Same as Option 1 but with per-row sanity prompt for DEAD-d candidates. Reduces false-DEAD risk but adds workflow weight. | |

**User's choice:** Add 4th criterion: 'dormant-by-ADR' (Recommended)
**Notes:** ADR-cite requirement is sufficient safety net; boundary check added inline in CONTEXT D-02 anyway ("prefer DEAD only when ADR section explicitly states dormancy with no activation trigger in current milestone").
