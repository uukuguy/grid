### DEAD definition (D-02)

**4 DEAD criteria** (row may match any one):

- **(a) octo-leftover**: row references `octo_*` / `octo:hook` / `OctoHookPlugin` / pre-Phase-BA naming that no longer exists in the codebase (cross-ref Phase BA 2026-04-04 commit).
- **(b) deleted-code**: row points at a file/module that has been removed (e.g., references to crates renamed away or files deleted in Phase 5.x refactors).
- **(c) ADR-superseded**: row's intent was nullified by an Accepted ADR — e.g., hermes-runtime rows after `ADR-V2-017` froze it; rows referencing v1 protocol after v2 ADRs; rows on backend that was replaced.
- **(d) dormant-by-ADR**: row depends on a capability that an Accepted ADR explicitly marks as dormant / deferred-indefinitely. Rationale **must cite the specific ADR section** asserting dormancy (e.g., `ADR-V2-024 §双轴 framework: grid-platform dormant`). Catches D62-64 (Sandbox Tiers — never built, V2-024 dormant), D108+ `grid-platform/grid-desktop/web*` rows.

**Boundary rule**: If a row matches both DEAD-(d) AND P3, prefer DEAD only when the ADR section explicitly states dormancy with no activation trigger in current milestone. Otherwise default to P3 (preserves audit trail).

**Anti-pattern**: Do NOT create a 5th criterion during triage. If unsure, classify P3 with rationale `unable to classify DEAD per 4-criterion test — kept in P3 for future review`.
