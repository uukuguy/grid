### Item 1: D74 (CONTRACT-06) — EmitEvent gRPC Reverse Channel

**Current state:**
- `proto/eaasp/runtime/v2/runtime.proto:76` defines `rpc EmitEvent(EventStreamEntry) returns (Empty)` inside the `Runtime` service.
- Current flow is L4→L1: L4 calls `EmitEvent` on L1's gRPC server (standard RPC pattern). L1's `service.rs:597-613` handles it by delegating to `RuntimeContract::emit_event()` (no-op by default).
- `RuntimeContract::emit_event` at `contract.rs:110-121` is a trait method with a default no-op implementation. T1 runtimes override it.

**What needs to change (reverse channel):**
- L1 becomes a gRPC **client** that pushes events to L4's gRPC **server**.
- L4 FastAPI lifespan must start a gRPC server (new `EventSink` service) listening on a new port.
- L1 runtime needs a gRPC client that calls `EmitEvent` when events are generated (PRE_COMPACT, tool use, etc.).
- The proto already has `EmitEvent` RPC and `EventStreamEntry` message — no new proto definitions needed for the message format, but a separate `EventSink` service definition is required (since gRPC doesn't allow a server to be both server and client for the same RPC in this pattern).

**ADR gate:** Must create ADR for `EventSink` service addition. The ADR establishes that `EmitEvent` is OPTIONAL per ADR-V2-001 — fallback is no-op if L4 not available.

**Relevant existing code:**
- `proto/eaasp/runtime/v2/runtime.proto:71-76` — EmitEvent RPC definition (current L4→L1 direction)
- `proto/eaasp/runtime/v2/runtime.proto:258-269` — EventStreamEntry message
- `crates/grid-runtime/src/service.rs:597-613` — current server-side EmitEvent handler
- `crates/grid-runtime/src/contract.rs:110-121` — RuntimeContract::emit_event (no-op default)
- `crates/grid-runtime/src/pre_compact_emitter.rs` — event emission to file (reference pattern for event generation)

**Implementation approach:**
1. ADR: create `EventSink` service definition in proto (or separate proto file)
2. Proto regeneration (`scripts/gen_runtime_proto.py`)
3. L4: add gRPC server to FastAPI lifespan
4. L1: add gRPC client, push events when generated
5. ~100-150 LOC total

### Item 2: D139 (CONTRACT-07) — 双 Terminate Semantics (Doc-Only)

**Current state:**
- The `TerminateRequest` RPC is defined at `proto/eaasp/runtime/v2/runtime.proto` (line ~27-29): `rpc Terminate(TerminateRequest) returns (Empty)`.
- Two termination paths exist: (1) normal via `TerminateRequest`, (2) timeout/error-imposed via the loop's natural termination.
- Contract currently doesn't specify whether a second `TerminateRequest` on an already-terminated session is NO-OP or an error.

**What needs to change:**
- Document the two paths in the contract spec.
- Specify enforcement expectations (NO-OP vs error).
- Add cross-runtime test to enforce consistent behavior across all 7 runtimes.
- ~20 LOC doc in ADR or ADR-V2-017 revision + cross-runtime test.

**Relevant existing code:**
- `proto/eaasp/runtime/v2/runtime.proto:27-29` — TerminateRequest (to be confirmed)
- `docs/design/EAASP/adrs/ADR-V2-017-hermes-retirement.md` — candidate for revision

**Implementation approach:**
- Doc-only + cross-runtime contract test
- NO-OP for second terminate is the simplest and most common convention

### Item 4: D130 (ENGINE-08) — Cancel Token Parent→Child Propagation Fix

**Current state — DUAL CODE PATHS:**

**Path A (OLD — `cancellation.rs`):**
- `ChildCancellationToken` at `cancellation.rs:72-81` has NO `cancel()` method — only `is_cancelled()`.
- The `CancellationToken` has `child()` at line 58 that returns `ChildCancellationToken`.
- Child reads parent's flag but cannot propagate its own cancellation.
- Tests at `cancellation.rs:118-128` show child inheriting parent cancellation, but no test for child→parent propagation.

**Path B (NEW — `cancellation_tree.rs`):**
- `CancellationTokenTree` at `cancellation_tree.rs:150` provides session/turn token hierarchy with shared `AtomicBool` flags.
- Already wired into `AgentExecutor` (field at `executor.rs:154`, used at line 269 and 371).
- `AgentMessage::Cancel` handler at `executor.rs:633-637` cancels the session token.
- `TurnToken::to_cancellation_token()` at line 133 creates a **snapshot** `CancellationToken` — it pre-cancels if session was cancelled BEFORE the call, but does NOT maintain a live link to the shared flag.

**The D130 bug (remaining issue):**
When `turn_token.to_cancellation_token()` is called at `executor.rs:567`, the harness receives a **disconnected** `CancellationToken`. If the session is cancelled AFTER this point (mid-turn), the harness loop's check at `harness.rs:695` (`config.cancel_token.is_cancelled()`) returns `false` — it won't see the cancellation until the NEXT turn when a fresh token is created.

This means `session kill` CLI command does NOT immediately interrupt in-flight LLM calls if the cancellation happens after the token snapshot was taken. The harness only sees it on the next iteration or next turn.

**What needs to change (per CONTEXT.md D-06/D-07):**
- Add `ChildCancellationToken.cancel()` method that propagates to the parent OR
- Fix `TurnToken::to_cancellation_token()` to return a token that checks the live shared flag on every `is_cancelled()` call instead of taking a snapshot.

**Fix approach (recommended):**
Modify `TurnToken::to_cancellation_token()` to return a cancellation token that checks live state:

```rust
pub fn to_cancellation_token(&self) -> CancellationToken {
    // Return a token whose is_cancelled() checks the LIVE shared flags
    // — not a snapshot. This way, mid-turn session cancellation is
    // immediately visible to the harness loop.
}
```

Alternative: Have the harness loop read `config.cancel_token` from the `TurnToken` directly instead of converting to `CancellationToken`. But this requires changing the `AgentLoopConfig` field type from `CancellationToken` to something that supports live-check.

**Files affected:**
- `crates/grid-engine/src/agent/cancellation_tree.rs` — fix `to_cancellation_token()`
- `crates/grid-engine/tests/` — add test for mid-turn cancellation

**~30-50 LOC (per CONTEXT.md D-08)**

## Recommendations

### Plan Structure (5 items → 3 plans, 2 waves)

**Wave 1 (parallel — no shared files):**
- **Plan 01 (contract):** D74 EmitEvent + D139 Terminate — both contract/proto work
- **Plan 02 (engine):** D106 MAX_TURNS + D130 cancel token — both grid-engine changes
- **Plan 03 (server):** D90 WS schema — verify-and-close

**Wave 2:**
- None needed — no inter-plan dependencies. ADR creation for D74/D139 can be embedded in Plan 01.

### Option A (recommended): Three parallel plans

**Approach:** Split by crate boundary — contract/proto (D74+D139), grid-engine (D106+D130), grid-server (D90).
**Why:** Zero file overlap between plans. D74 proto work is independent from engine config fixes. D90 is likely verification-only.
**Cost:** Low — all items are well-understood with existing patterns.

### Option B: Two plans (contract + engine)

**Approach:** Merge D90 into either plan.
**Why not:** D90 might be already done — verify-and-close in a separate plan is cleaner.

### Option C: Single plan

**Approach:** All 5 items in one plan.
**Why not:** Different crates, different testing patterns. Context cost too high for one plan.
