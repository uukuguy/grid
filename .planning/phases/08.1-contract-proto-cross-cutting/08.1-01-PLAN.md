  <action>
  Revise ADR-V2-017 §2 (or create an addendum section) to clarify the two Terminate paths per CONTEXT.md D-11/D-12.

  **Per LEDGER L251**, two behaviors need specification:

  1. **Path 1 — Normal Terminate via `TerminateRequest` RPC**: Session is in active/initialized state → `TerminateRequest` triggers graceful shutdown → returns `Empty` (OK). Subsequent `TerminateRequest` on already-terminated session → **NO-OP** (returns `Empty`, not `FAILED_PRECONDITION`). Rationale: idempotent terminate is safer for retry-heavy orchestrators; L4 should already know session state.

  2. **Path 2 — Timeout/error-imposed termination**: Session encounters timeout, unrecoverable error, or external cancel → session enters terminated state via internal mechanism (not `TerminateRequest`). Subsequent `TerminateRequest` on such a session → also NO-OP (same idempotent guarantee).

  **Canonical error code for unknown session**: When `Send` is called on a session_id that was never initialized → return `NOT_FOUND` (not `FAILED_PRECONDITION`). This is the gRPC-standard code for "resource does not exist."

  **Cross-runtime enforcement**: Add a contract test in `tests/contract/test_e2e_smoke.py` (or new file `test_terminate_semantics.py`) that:
  - Initializes session → Terminates → Terminates again → asserts second call returns OK (NO-OP)
  - Sends to never-initialized session_id → asserts `NOT_FOUND` error code
  - Run across all 7 runtimes to verify consistency

  Doc change only (~20 LOC doc). No code implementation for the ADR itself — ADR defines the contract; enforcement test is in this task.
