## Findings

### F1: Send-path test is one indirection removed from the spec wording [WARNING]

**Where:** `crates/grid-runtime/tests/harness_payload_integration.rs:158-237` (commit `60f99d5`)
**Dimension:** test-quality

**Finding:**
The plan's T03 Step 3 specified: "drive a Send call and assert the preamble survives into the request body" — with a fallback authorization to add a `#[doc(hidden)] pub fn` test-only accessor on `GridHarness` if `session_store` could not observe history without a live LLM. The executor took a different route: extracted the inline body of `initialize()` into a new `pub fn build_initial_history_for_payload`, then had the test call that same function on the payload BEFORE calling `initialize()` and assert on its output.

This is NOT a pure tautology — the helper IS exercised end-to-end (the `assert_eq!`s on the snapshot prove the helper actually produces a System-role `ChatMessage` with the preamble markers for a realistic payload fixture). But the assertion shape is `assert(helper(payload) == expected)`, while `initialize()` also calls `helper(payload)`. There is no assertion that crosses `initialize()` itself — if someone refactored `initialize()` to bypass `build_initial_history_for_payload` (e.g. inlined a different path), the test would still pass because it never observes the `Vec<ChatMessage>` that actually reached `start_session_with_tool_filter`. The historical D58 concern ("`test_initialize_injects_memory_refs_preamble` 未走 Send 全路径") is technically still partially open in spirit.

SUMMARY §Deviations #3 explicitly acknowledges this and explains the no-live-LLM constraint. The deviation is defensible — but it is a deviation, and the LEDGER L178 closure narrative does not flag the residual gap.

**Suggested fix:**
Either (a) add a one-line `assert!` on the result of `initialize()` itself that confirms a non-error handle is returned (the existing `expect("initialize must succeed")` at L168 already does this, so verify it's actually load-bearing for the preamble path), OR (b) add a small follow-up note to the D58 LEDGER row narrative that says "shared-helper observation; full session_store read-back deferred until integration harness gains a mock LLM" so the partial coverage is on the record. No code rewrite required; this is a docs-clarity polish.

### F2: T01/T02 strict-by-default semantic is correct but the rationale is not in the struct rustdoc [NIT]

**Where:** `crates/grid-engine/src/context/compaction_pipeline.rs:44-45` (commit `3a23ecb` final state)
**Dimension:** strict-by-default

**Finding:**
The plan T01 Step 1 specified `#[serde(deny_unknown_fields)]` only. The executor widened to `#[serde(deny_unknown_fields, default)]` in T02 because a partial `compaction:` block (e.g. only `proactive_threshold_pct: 60`) would otherwise require all 10 fields. The semantic is correct — verified by the 3/3 PASS of `compaction_yaml_wiring.rs` including the `unknown_compaction_key_rejects_per_adr_v2_028` canary which confirms unknown keys still reject — and this matches ADR-V2-028 intent ("tolerate omissions, reject typos"). The deviation is documented in SUMMARY §Deviations #1.

However: the struct's own rustdoc block at L31-45 explains each field but does NOT document the strict-by-default contract at the struct level. A future engineer reading the struct alone (without SUMMARY) sees `#[serde(deny_unknown_fields, default)]` with no anchor to ADR-V2-028 or D102. The grid-server `Config.compaction` field at `crates/grid-server/src/config.rs:90-95` DOES have the right doc-comment ("ADR-V2-028 strict-by-default") — but the inner struct on which the attribute actually lives does not.

**Suggested fix:**
Add a 2-line rustdoc paragraph above `#[derive(...)]` at `compaction_pipeline.rs:43` (or extend the existing block above L31) that says something like: "Strict-by-default per ADR-V2-028: `#[serde(deny_unknown_fields, default)]` rejects unknown YAML keys (typo guard) while allowing partial blocks to fall back field-by-field to `Default::default()`." This is a 30-second edit, no behavior change. Can be deferred or skipped — the test names already make the contract obvious — but the inconsistency with the outer struct's doc style is worth a NIT.

### F3: LEDGER closeout puts `Resolution:` in trailing column instead of a separate row [NIT]

**Where:** `docs/design/EAASP/DEFERRED_LEDGER.md` L106, L177, L178, L232, L233, L234 (commit `2852bec`)
**Dimension:** ledger

**Finding:**
The plan T02 Step 10 (and parallel steps in T03/T04/T05/T06) quoted a closeout template `✅ CLOSED YYYY-MM-DD Phase X.X Plan NN @ <commit>` followed by "**Resolution**: <narrative>" on the next line. The actual implementation puts the closeout marker in the row's status column and the resolution narrative in the trailing notes column on the SAME table row, e.g.:

```
| **D57** | ... | ✅ CLOSED 2026-06-01 Phase 7.0 Plan 01 @ `60f99d5` | **Resolution**: Promoted ... |
```

This matches the existing single-row format of the LEDGER table (each D-row is one markdown table row, columns separated by `|`). The plan's quoted template would have implied a multi-line narrative block — but a markdown table cell cannot easily hold multi-line content without breaking the table. The executor's choice keeps the table parseable and matches recent precedent (D60 at L227 uses the same pattern).

This is purely a stylistic NIT — the format is auditable, contains all required information (date, phase, plan, commit hash, resolution narrative), and matches the table's structural constraints. No fix needed; flagging only because the plan's template wording could be tightened in future phases to say "trailing-column" rather than "next-line" to avoid ambiguity.

**Suggested fix:**
None required for this phase. For future plan authors: when quoting LEDGER closeout templates, say explicitly "in the trailing notes column" rather than "followed by" to avoid the multi-line/single-cell ambiguity.
