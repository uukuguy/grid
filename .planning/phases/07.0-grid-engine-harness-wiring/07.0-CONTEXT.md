<decisions>
## Implementation Decisions

### D-01 — D102 YAML wiring scope: minimum viable path
**Decision:** Implement the smallest path that closes D102 substantively:
1. Add `Serialize, Deserialize` derives to `CompactionPipelineConfig` at `crates/grid-engine/src/context/compaction_pipeline.rs:43` (currently `#[derive(Debug, Clone)]` only).
2. Add `pub compaction: CompactionPipelineConfig` field to `grid-server` `Config` struct at `crates/grid-server/src/config.rs:39-39+` with `#[serde(default)]` (matches the existing precedent for `server`, `provider`, `database`, `logging`, etc.).
3. Wire forward at the runtime-factory call site: locate where `AgentLoopConfigBuilder` is constructed inside grid-server (likely `crates/grid-server/src/api/agents.rs` per LEDGER L232 hint) and thread `cfg.compaction.clone()` into `.compaction_config(...)`.
4. Strict-by-default per ADR-V2-028: unknown YAML keys under `compaction:` reject (no silent drop); missing keys → use `CompactionPipelineConfig::default()` per existing default impl at compaction_pipeline.rs:69-83.
5. Round-trip test: write `compaction:` block → load via grid-server config loader → assert struct values match expectation.

**Out:** No `GRID_COMPACTION_*` env-var override surface; no `grid cli-config validate` echo for compaction. Both deferred (P3-ish, can be added in a v3.4+ config-doctor pass if user demand surfaces).

**Why deferred wiring is not chosen:** Splitting "add serde derives + Config field" from "thread into factory" creates a transient state where YAML parsing accepts the block but runtime ignores it — confusing for users hitting it mid-milestone. The minimum-viable path closes D102 atomically.

### D-02 — D57 DRY fix: promote `build_memory_preamble` to `pub`
**Decision:** Change `pub(crate) fn build_memory_preamble` at `crates/grid-runtime/src/harness.rs:144` to `pub fn`. Update `crates/grid-runtime/tests/harness_payload_integration.rs` to import `GridHarness::build_memory_preamble` directly and delete the copied `expected_preamble` at line 470-485. Single source of truth.

**Why not parity test:** Adds permanent maintenance burden (two implementations must stay in sync forever). Visibility change is a 1-LOC fix with zero downside — `build_memory_preamble` is already a pure helper with no internal-state dependency.

**Why not move to grid-types:** Out of proportion. The helper is grid-runtime-specific (formats memory_refs for runtime preamble injection); grid-types is for shared type definitions only (zero-dep crate).

### D-03 — D58 Send-path test coverage: extend existing test
**Decision:** Augment `test_initialize_injects_memory_refs_preamble` in `harness_payload_integration.rs` to additionally drive a Send call (after Initialize) and assert the memory preamble survives into the request body. One additional assertion block at the end of the existing test, not a separate test file.

**Why:** Keeps the test's narrative intent (Initialize → preamble shows up in subsequent dispatch). After D-02 lands, the cross-crate test will already be using the public `build_memory_preamble` helper, so this assertion gets clean shared fixture data for free.

### D-04 — D103 boundary scan: profile-decide-record
**Decision:** Build a small benchmark in `crates/grid-engine/tests/` (e.g. `tests/find_tail_boundary_perf.rs`) that constructs synthetic 200-turn conversations at varying `tail_protect_tokens` (10k / 50k / 200k), measures wall-clock for `find_tail_boundary` invocation, and reports as a one-shot pytest-style assertion (e.g. "p99 under 50ms at 200-turn × 200k tail"). Then decide: if measured baseline is acceptable → close LEDGER row with "measured baseline OK, deferred refactor; rustdoc complexity note added"; if regression risk surfaces → file a follow-up D-row for v3.4+ refactor. Either way, ENGINE-05 closes with measurement evidence in LEDGER close-out.

**Why not eager fix:** LEDGER L233 explicitly says "current implementation is O(sum of msg sizes), not true O(N²); only hot in extreme long-session + large tail_protect_tokens cases". Refactoring without measured evidence is speculative.

### D-05 — D104 reactive guard placement: keep in harness, doc-only LEDGER close, no ADR
**Decision:** Do not move `attempted_reactive_compact` guard from harness to pipeline. Add rustdoc explaining the invariant ("a single agent loop turn can trigger reactive compaction at most once") near the existing guard call site, and close LEDGER row L234 with verdict "no other caller of `CompactionPipeline::compact()` exists today outside the harness; per ADR-V2-018 §D5 the per-turn semantic is preserved; deferred until a second caller emerges". No new ADR drafted.

**Why no ADR:** LEDGER L234 already records that the current state is ADR-V2-018 §D5 compliant. Drafting ADR-V2-030 candidate for a re-architecture that isn't happening adds documentation debt without enforcement value. If a second caller emerges in v3.4+ this can be revisited.

### D-06 — Plan structure: single plan, 5-6 tasks
**Decision:** All ENGINE-01..06 land in one plan file `07.0-01-PLAN.md`. Approximate task layout:
- T01 — Add serde derives to `CompactionPipelineConfig` (D-01 step 1).
- T02 — Add `compaction` field to grid-server `Config` + factory wiring + strict-by-default test (D-01 steps 2-5).
- T03 — Promote `build_memory_preamble` to `pub` + delete test copy (D-02) + extend test for D58 Send path (D-03).
- T04 — `find_tail_boundary` benchmark + LEDGER close + rustdoc complexity note (D-04).
- T05 — D104 doc-only close: rustdoc invariant note + LEDGER row close-out (D-05).
- T06 — ENGINE-02 no-op close: LEDGER row D3 close-out with "deferred per scope ceiling, see Phase 7.0 CONTEXT D-domain out-of-scope".

**Why not split:** 6 cohesive tasks all touch the same module (`crates/grid-engine/src/`); splitting creates artificial PR seam. ROADMAP §Notes already implies ≤6 row scale.

## Existing Code Insights

### Reusable Assets
- **Config precedent**: `crates/grid-server/src/config.rs:39-80` `Config` struct has clean precedent — each subsection is a field with `#[serde(default)]`. Adding `compaction: CompactionPipelineConfig` follows this verbatim. No new abstraction needed.
- **AgentLoopConfigBuilder.compaction_config()**: Builder method already exists at `crates/grid-engine/src/agent/loop_config.rs:457` taking `CompactionPipelineConfig` by value. The factory just needs to call it.
- **CompactionPipelineConfig::default()** at `compaction_pipeline.rs:69-83`: clean defaults (proactive_threshold_pct=75, tail_protect_tokens=20_000, reactive_only=false, etc.) — `#[serde(default)]` on the Config field automatically picks this up when YAML omits the section.
- **GridHarness::build_memory_preamble** at `harness.rs:144`: pure function with no `self`/internal-state dependency — visibility promotion is safe.

### Established Patterns
- **Strict-by-default config validation (ADR-V2-028 lineage from Phase 5.4)**: existing config sections reject unknown keys via serde's `deny_unknown_fields` or equivalent. Replicate the same pattern on `CompactionPipelineConfig` if it isn't applied at the parent `Config` level.
- **LEDGER close-out style**: matches Phase 5.5 / 6.x — append `✅ CLOSED YYYY-MM-DD Phase X.X Plan NN @ commit-hash` + 1-2 sentence resolution paragraph + commit + verification hint. See L499 (NEW-L1 close-out) for the gold-standard format.
- **Pure const + rustdoc complexity note**: existing pattern at `harness.rs:122-123` (MAX_TURNS_FOR_BUDGET) — multi-line `///` doc explaining purpose + tradeoff. Mirror this style for D-04 rustdoc addition.
- **No `unsafe` and no test-only `#[cfg(test)]` cross-module shims**: per CONVENTIONS.md `unsafe`=zero; per CLAUDE.md test discipline, the D-02 fix is a real visibility change, not a `#[cfg(test)] pub` workaround.

### Integration Points
- **Grid-server runtime factory** (D-01 step 3): the precise call site that builds `AgentLoopConfig` for live sessions in grid-server. LEDGER L232 hints `crates/grid-server/src/api/agents.rs` + engine agent factory; planner should grep for `AgentLoopConfigBuilder::new` and `AgentLoopConfigBuilder::build` to confirm.
- **Test fixture sharing** (D-02 + D-03): once `build_memory_preamble` is `pub`, the cross-crate test in `harness_payload_integration.rs` shares the same memory_refs formatter as the in-crate tests at `harness.rs:1262-1295`. No fixture-data duplication.
- **CompactionPipelineConfig as part of round-trip test**: round-trip test in D-01 step 5 should serialize then deserialize via `serde_yaml` (or whatever grid-server config loader uses) and assert all 10 fields survive. This is the canary that `Serialize` derive doesn't drop anything.
