## Implementation Decisions

### D-01 — D137 scope: all 3 sub-tasks, close 10/10 xfails
**Decision:** Land D137 in full — multi-turn session replay framework + MCP subprocess fixture + scripted PreCompact threshold. Closes 10/10 xfails across `tests/contract/contract_v1/test_event_type.py` (4 — CHUNK stream / terminal STOP / non-terminal chunk seq / ERROR shape), `test_mcp_bridge.py` (5 — real stdio subprocess / tool invocation / disconnect / reconnect / error), `test_proto_shape.py:test_events_stream_emits_stop_at_turn_end` (1).

**Sub-task breakdown (planner reference, not binding):**
- **Multi-turn session replay framework**: extend `tests/contract/harness/` to support recorded session fixtures (sequence of turns with deterministic provider responses); session_id reuse across turns; per-turn assertion hooks. Reuse existing `mock_openai_server.py` + `mock_anthropic_server.py` fixtures where possible.
- **MCP subprocess fixture**: spawn real stdio MCP server subprocess (likely `eaasp-l2-memory-engine` or a thin test MCP server) for the 5 `test_mcp_bridge.py` cases; manage lifecycle (start / kill on test exit / timeout). Reuse `tests/contract/harness/` patterns.
- **Scripted PreCompact threshold**: replace any hard-coded PreCompact threshold with a ChunkType-triggered emission path. `PRE_COMPACT = 8` enum already exists at `proto/eaasp/runtime/v2/runtime.proto:246`; `emit_telemetry` + `emit_event` are wired in `crates/grid-runtime/src/service.rs:453/597`. The work is wiring the threshold logic, not adding emit infrastructure.

**Why all 3:** ROADMAP success criterion #1 explicitly names all three. Partial close would leave D137 row only half-resolved in LEDGER. Phase 7.0 already wired the `CompactionPipelineConfig` YAML layer (commit `3a23ecb`), making the PreCompact threshold scriptable via config; building on that infrastructure right after is the natural sequence.

### D-02 — D138 mock approach: extend existing mock servers with scenario routing
**Decision:** Take LEDGER L250 option (a). Extend `tests/contract/harness/mock_openai_server.py` and `tests/contract/harness/mock_anthropic_server.py` to support:
1. `tool_choice` field in requests (required for `required_tools` enforcement).
2. Scenario-routed deny response — route by an HTTP header (e.g., `X-Test-Scenario: deny-non-required-tool`) to pre-configured fixture responses (e.g., "LLM called tool C when required_tools=[A,B]"; runtime should deny).

Closes 5/5 D138 xfails in `tests/contract/contract_v1/test_skill_workflow.py`.

**Why not gRPC bypass (option b):** Bypassing the LLM tests the enforcement layer in isolation but does NOT test the LLM-call → deny round-trip that production traffic actually takes. Scripted mock covers more semantic surface and reuses existing fixture infrastructure already battle-tested across 7 runtimes.

**Why not both:** Marginal coverage gain (gRPC bypass only adds a unit-level view of the same logic); not worth the maintenance burden.

### D-03 — D5 + D6 coupling: verify D5 migrated, add D6 SessionPayload assertions
**Decision:** D5 and D6 are coupled per LEDGER L109 ("等 D3/D5 决策后一并"). Approach:

**D5 close (verify-not-rewrite):** Run `rg "telemetry_v1" -t rust crates/ tests/` and confirm 0 production code references (scout already found only a stale comment in `crates/grid-runtime/tests/harness_payload_integration.rs:4`). Inspect `crates/grid-runtime/tests/grpc_integration.rs` directly; if the test already uses v2 envelope (which scout suggests is the case), close D5 LEDGER L108 with verdict "verified already migrated; no code change required; rg evidence cited". If even one production reference exists, file as a small follow-up task; do NOT punt to v3.4+.

**D6 close (concrete additions):** Extend `tools/eaasp-certifier/src/blindbox.rs` (`SessionPayload` builder at line 105) and `tools/eaasp-certifier/src/verifier.rs` (`SessionPayload` consumer at line 309) to assert each P1-P5 field is present and well-shaped. Add a `SessionPayload schema:` line to the certifier report so `make verify-dual-runtime` runs surface the assertion outcome.

### D-04 — D55 cross-runtime parity test
**Decision:** Add absence-path parity tests across Python + Rust + TS for proto3 submessage presence (`HasField` semantics). Concretely:
- Python: in `tests/contract/harness/` or per-runtime test, assert `HasField('field')` returns False on default-constructed messages and True after field assignment for each proto3 submessage in the contract.
- Rust: tests assert `Option<SubMessage>::is_some()` vs `is_none()` patterns are used consistently (NOT truthy fallback like `if msg.field` which is always-true for non-empty default sub-message values).
- TypeScript (`lang/ccb-runtime-ts/`): equivalent test using TS proto3 library's presence check.

Closes D55 LEDGER L175 with mechanically enforced consistency across the 3 language sides.

**Why not doc-only:** Without a test, the truthy-fallback anti-pattern can silently re-emerge. A CI-enforced parity test prevents regression at zero ongoing maintenance cost.

**Why not defer:** D55 is small (3 small additions to existing parity tests); deferring this and then having to re-establish context next milestone is more expensive than just doing it now.

### D-05 — Plan structure: single plan, 8-10 tasks
**Decision:** All 5 REQ-IDs land in one PLAN file `07.1-01-PLAN.md` with approximately:
- **T01** — Multi-turn session replay framework (D137 part 1)
- **T02** — Scripted PreCompact threshold (D137 part 2)
- **T03** — MCP subprocess fixture (D137 part 3)
- **T04** — Mock-server scenario routing extension (D138 infrastructure)
- **T05** — D138 deny-path test wiring (close 5 xfails in test_skill_workflow.py)
- **T06** — D5 verify + LEDGER close (with rg evidence)
- **T07** — D6 SessionPayload P1-P5 assertions in certifier
- **T08** — D55 cross-runtime HasField parity tests
- **T09** — LEDGER consolidation + commit-hash backfill (mirrors Phase 7.0 T07 pattern)

**Why not split by language:** Although D137/D138 are Python-heavy and D5/D6/D55 are Rust/TS-heavy, the contract observability story is a single coherent product change. Plan splitting just to honor language boundaries creates artificial PR seams. Phase 7.0 single-plan precedent worked well.

**Plan-level dependencies (intra-plan order):**
- T01 → T02 (T02's PreCompact test needs multi-turn framework)
- T03 standalone but should follow T01 (reuse harness patterns)
- T04 → T05 (T05's deny tests need T04's scenario-routed mock)
- T06 → T07 (T07 builds on D5 verification outcome)
- T08 independent of all above
- T09 last (consolidation)
