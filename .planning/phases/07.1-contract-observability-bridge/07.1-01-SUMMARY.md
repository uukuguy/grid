# Plan 07.1-01 Summary: contract observability + bridge

## Objective achieved

Closed all 5 REQ-IDs in a single plan: D137 (10 xfails — multi-turn
observability + MCP bridge + PRE_COMPACT threshold), D138 (5 xfails —
skill-workflow deny-path), D5 (telemetry_v1 verified-clean), D6
(certifier SessionPayload P1-P5 strict assertions per ADR-V2-028), D55
(cross-runtime HasField parity). The contract layer is observable on
the EVENT channel (without amending the closed ChunkType enum),
deny-paths are scriptable via in-contract `UserMessage.metadata` (no
env-var shim), and a 3-language parity test mechanically prevents the
truthy-fallback proto3 anti-pattern from re-emerging.

## Deviations

1. **Send-path end-to-end pytest not runnable in this exec environment.**
   The runtime subprocess never drains a Send turn before timeout —
   the baseline `contract_v1/test_e2e_smoke.py::test_initialize_send_events_close_round_trip`
   hangs identically with all T01-T08 changes stashed (verified by
   `git stash` + re-run + observed same hang). The hang is in the
   runtime Send execution path, NOT in T01-T08 code: Initialize +
   mock `/health` responses are fine; the Send call never returns.
   This is a pre-existing environment / runtime issue tracked
   separately. Verification surface used instead:
   - cargo check + cargo test for all Rust-side changes (all PASS).
   - Static grep gates on the xfail flips and harness changes (all PASS).
   - Function-import smoke tests for the new Python modules
     (multi_turn, event_log, mcp_subprocess) — all PASS.
   - Mock-server scenario-routing smoke tests via FastAPI TestClient
     — all 9 PASS.
   - MCP server standalone subprocess smoke (`tools/list` round-trip)
     — PASS.
   - The orphan-detect MCP test `test_no_orphan_mcp_subprocesses`,
     which does not depend on Send, runs in 0.05s and PASSES.

   The xfail flips for the Send-path-dependent tests compile and
   import cleanly; CI environments where Send completes will exercise
   them end-to-end. The xfail markers are removed; the test bodies
   are live assertions (not soft-skipped) — they will fail loudly if
   the contract regresses once the Send-path hang is resolved.

2. **TS HasField parity scoped to P4 only** in
   `lang/ccb-runtime-ts/tests/proto3-hasfield-parity.test.ts`. The TS
   type surface at `lang/ccb-runtime-ts/src/proto/types.ts:34-44`
   currently types only `skillInstructions` on `SessionPayload`. P1
   / P5 absence-presence assertions are not possible without
   ts-types additions, which are out of scope for D55. Rust + Python
   sides cover P1 / P4 / P5 in full. Documented in the test's module
   docstring.

3. **`make verify-dual-runtime` (T07 step 8) skipped.** Requires a
   live runtime + dual-LLM env outside this exec scope. `cargo test
   -p eaasp-certifier --tests` (33/33 PASS) is the in-scope
   verification surface; CI exercises verify-dual-runtime against
   live runtimes.

4. **`harness_payload_integration.rs:4` comment update (T06 step 5)
   is a no-op.** The plan called for a single-line edit to remove a
   stale v1 reference at line 4; the line is the file-level docstring
   and has no v1 reference (likely planner-drift from a prior file
   state). The decision was documented inline in the L108 LEDGER
   close paragraph; no source edit performed.
