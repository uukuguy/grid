### F1: LEDGER L249 over-claims `emit_event` RPC path for PRE_COMPACT — actual implementation is a file-sink hook [WARNING]

**Where:** `docs/design/EAASP/DEFERRED_LEDGER.md` L249 (D137 closeout, commit `d7cd524`) vs `crates/grid-runtime/src/pre_compact_emitter.rs:103-157` (commit `a52b2c4`)
**Dimension:** T02 EVENT-channel correctness / documentation accuracy

**Finding:**
The plan PLAN.md T02 Step 2 specified: "register a PreCompact hook handler that — when fired — calls the runtime's contract `emit_event` with an `EventStreamEntry { ... event_type: HookEventType::PreCompact, ... }`." The LEDGER L249 D137 closeout narrative reflects that wording verbatim:

> "(T02 @ `a52b2c4`) wired PRE_COMPACT emission via the EVENT channel (`HookEventType::PRE_COMPACT = 8` → `EventStreamEntry` → existing `emit_event` RPC at `service.rs:597`) when `CompactionPipelineConfig.proactive_threshold_pct` is breached."

The actual implementation in `pre_compact_emitter.rs:103-157` does NOT call `RuntimeContract::emit_event` and does NOT construct an `EventStreamEntry`. It writes a JSON object literal `{"event_type": "PRE_COMPACT", ...}` directly to `${GRID_CONTRACT_PROBE_OUT}/events.jsonl` via `std::fs::OpenOptions::new().append(true)`. The `emit_event` RPC at `service.rs:597` is the **inbound** RPC handler (L4 → L1) — `grep -rn 'emit_event(' crates/grid-runtime/src/` shows the only outbound call is `RuntimeContract::emit_event(...)` invoked from inside that handler; nothing in the T02 PRE_COMPACT path crosses that wire.

The architectural intent (PRE_COMPACT does NOT use the CHUNK channel) is fully preserved — anti-canaries are clean. The `pre_compact_emitter.rs:17-28` rustdoc explicitly documents the file-sink choice with rationale: "an in-memory channel would require new RPC plumbing or a new HTTP endpoint inside grid-runtime, both of which are heavier than reusing the existing `GRID_CONTRACT_PROBE_OUT` directory". So the implementation choice is defensible.

The discrepancy is therefore between (a) the LEDGER L249 narrative + the SUMMARY T02 bullet "Hook handler ... appends a `{"event_type": "PRE_COMPACT", ...}` JSON line" (which IS accurate) and (b) the same LEDGER L249 paragraph also claiming the wire path is `EventStreamEntry → emit_event RPC at service.rs:597` (which is NOT what the code does). Future readers diffing the LEDGER against actual code will see the contradiction.

**Suggested fix:**
Edit the LEDGER L249 D137 narrative to replace "`HookEventType::PRE_COMPACT = 8` → `EventStreamEntry` → existing `emit_event` RPC at `service.rs:597`" with something like: "`HookEventType::PRE_COMPACT = 8` (no chunk variant added) observable via the `PreCompactEmitter` HookHandler that writes `{"event_type": "PRE_COMPACT", ...}` JSON lines to `${GRID_CONTRACT_PROBE_OUT}/events.jsonl` for contract-test consumption — the EVENT-channel `emit_event` RPC plumbing remains untouched". This keeps the anti-canary claim (no chunk channel) front-and-center while accurately reporting the observation surface. No code change required.

### F2: T05 `test_scenario` cell is process-global, not session-scoped despite the function name [WARNING]

**Where:** `crates/grid-engine/src/providers/test_scenario.rs:37-50, 61-73` (commit `e2460df`)
**Dimension:** T05 strict-by-default + concurrency correctness

**Finding:**
The module rustdoc at lines 24-35 says "Threading model: Process-global `RwLock<HashMap<session_id, String>>`. The set/clear pair brackets the executor's Send turn." The implementation at lines 40-45 actually defines `ScenarioState { current: Option<String> }` — a single `Option<String>` cell, NOT a `HashMap` keyed by session_id. Lines 32-35 partly acknowledge this: "Since the OpenAI provider does not know the session id at request-build time, we shortcut to a 'current scenario' cell that is set/cleared around the Send call." The public API names — `set_session_scenario` / `clear_session_scenario` / `current_session_scenario` — all carry "session" in the identifier, which suggests per-session isolation that the implementation does not provide.

If two contract tests drove `runtime_grpc_stub.Send(...)` concurrently against two different sessions with two different `x-test-scenario` metadata keys, the harness's `set_session_scenario` calls would race against the single `current` cell, and the OpenAI provider could read either value. The plan/CONTEXT did not require concurrent-session scoping, and the contract suite serializes test execution by default, so the bug is latent — but the naming invites a future caller to assume safety that does not exist.

Bounded impact: the cell only emits a `X-Test-Scenario` outbound HTTP header on the LLM-call side, and only when the metadata key is set (production callers never set it). No data corruption, no live-traffic side effect. But "you can't tell from the API whether it's session-scoped or process-scoped" is a maintenance hazard.

**Suggested fix:**
Either (a) rename the API to `set_current_scenario` / `clear_current_scenario` / `current_scenario` to match what the implementation actually does, OR (b) keep the names and document the concurrency caveat at the top of `set_session_scenario`'s rustdoc with a one-line "NOTE: process-global cell; concurrent Send turns from different sessions WILL race. Contract suite serializes tests so this is currently safe." Either edit is <5 lines, no behavior change. Defer or address opportunistically.
