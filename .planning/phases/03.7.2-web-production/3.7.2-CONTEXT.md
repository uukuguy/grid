# Phase 3.7.2 — web/ dashboard 实战可用性 — Context

## Phase Boundary

`web/` (Grid single-user workbench UI, leg B per ADR-V2-024) must let a non-developer observer open the dashboard, observe a running agent, see its tool calls, see its memory writes, and stop/resume it **without devtools intervention**. Two-plan structure: Plan 01 audits gaps and produces REQ-WEB-NN backlog; Plan 02 closes the gaps. Acceptance: self-recorded walkthrough of a non-developer observer running S3 hook governance scenario end-to-end through `web/`.

**Activation state:** `web/` scored **9.0/10** at Activation audit (8 tabs, 9 vitest tests, "no mocks" — confirmed by `grep TODO|FIXME|mock|stub` returning 0 matches). Phase 3.7.2 is NOT a quality score bump — it's the gap between code quality (9.0) and **实战可用** (non-developer can drive the agent).

## Implementation Decisions

### D-01 — Plan 01 scope (locked 2026-07-20)

**Audit + spec scenarios** approach (mirrors Phase 3.7.1 Plan 01):

Plan 01 produces `docs/audit/3.7.2-GAP-AUDIT.md` with:
- 8-page surface matrix: Chat / Collaboration / Debug / Memory / Schedule / Tasks / Tools / McpWorkbench
- 5-event-type matrix per page: open / observe running agent / see tool calls / see memory writes / stop-resume
- Per-cell verdict: PASS / MANUAL TUNING / FAIL
- Per-FAIL row evidence (file:line) + REQ-WEB-NN derived (target: ≤ 9 REQ-WEB items, similar to Phase 3.7.1's 9 REQ-AUDIT cap)
- No code changes — read-only investigation. ~1-2 days.

Plan 02 closes the backlog. If findings exceed 1 plan, defer overflow to Plan 03.

### D-02 — UX surface: SessionBar as global controller (locked 2026-07-20)

Promote `web/src/components/SessionBar.tsx` (currently 6.3K) to a global controller with:

| Affordance | Source of truth | Backend action |
|---|---|---|
| **Stop button** (visible when session.status === "running") | `tasks[].status` atom | `POST /api/v1/sessions/:id/kill` (mirrors `grid session kill`) |
| **Resume button** (visible when session.status === "paused" or "stopped") | session.status atom | `POST /api/v1/sessions/:id/resume` (mirrors `grid session resume`) |
| **Live indicator** (pulsing dot during streaming) | `isStreamingAtom` | Server WS `text_delta` events |
| **Memory write toast** (transient toast on `memory_added` event) | new `pushMemoryEventAtom` | Server WS broadcast on L2 memory write |

**Position**: cross-page fixed bottom-right (already in SessionBar's layout pattern; verify position rules in `layout/`).

**Rationale**: SessionBar already exists and is loaded in the root layout. Promoting it avoids adding a new global component; minimises layout changes.

### D-03 — Validation strategy (locked 2026-07-20)

Three-layer validation:

1. **Vitest unit tests** (extend existing 9-test base): cover SessionBar Stop/Resume state transitions, `pushMemoryEventAtom` toast emission, `wsManager` reconnect backoff logic.

2. **Playwright e2e tests** (NEW — use existing `.playwright-cli/` directory): full UAT scenarios:
   - E1: Open dashboard → Tasks page → see "running" task → click Stop → verify UI updates + backend POST fires
   - E2: Trigger memory write from running agent → verify toast appears + Memory page updates
   - E3: Kill server mid-session → verify WS auto-reconnect with session_id param

3. **Self-recorded walkthrough** (user as non-developer observer): record screen while running `grid quickstart S3` (hook governance scenario) end-to-end through `web/`. Verify: see tool calls live, see memory writes, stop agent mid-stream, resume cleanly, no devtools intervention needed.

### D-04 — Tool-call event ordering (locked 2026-07-20)

**Verify existing ordering + add sequence number for debug visibility**. No Lamport clock redesign (scope creep).

Concretely:
- `ServerMessage` adds optional `seq: u64` field (per `ws/types.ts`). Server increments per message.
- UI logs sequence gaps in debug mode (visible only when `?debug=1` query param set).
- Existing event ordering preserved (no reorder, no buffering).

### D-05 — WS reconnect stress test (locked 2026-07-20)

Existing `ws/manager.ts` has `maxReconnectAttempts = 5` + exponential-style backoff (timing TBD in audit). Verify under "agent crash" scenario:

- Test: kill server during active streaming → client should reconnect with `session_id` query param → resume from last sequence.
- Vitest: simulate `WebSocket.onclose` 5 times, verify reconnect attempts increment, verify 6th attempt gives up cleanly.
- E2E (Playwright): start server, open dashboard, kill server, restart server, verify WS re-establishes within 30s.

### D-06 — Mock fallback audit (locked 2026-07-20)

ROADMAP lists "mock fallbacks still in code paths per A.2 audit" as a candidate gap. Code grep across `web/src/` for `TODO|FIXME|mock|stub` returns **0 matches** (confirmed at discuss-phase time). Audit doc will record this as N/A with evidence. If audit Plan 01 finds mocks in any of the 8 pages, promote them to REQ-WEB-NN items.

### D-07 — Walkthrough deliverable format (locked 2026-07-20)

Self-recorded walkthrough produces:
- 1 markdown file `docs/status/WEB_PRODUCTION_USABILITY_<date>.md` with step-by-step timestamps
- 1 screen recording (`.webm` or `.mp4`, hosted in `.planning/phases/03.7.2-web-production/walkthrough/`)
- 1 markdown summary `docs/cli/scenarios/S7-web-dashboard.md` mirroring S1-S6 pattern (cross-link from QUICKSTART)

### D-08 — Inherits parent constraints (locked 2026-07-20)

All decisions from Phase 3.7.1 parent apply:
- D-04 (inherited): Test coverage must be hermetic (no LLM API key required for UAT)
- D-07 (inherited): TTY-aware JSON output (applies to WebSocket events: JSON envelope for non-streaming endpoints, JSON-line for SSE streaming)

## the agent's Discretion

- **Sequence number field name** (`seq` vs `n` vs `message_id`): `seq` is recommended.
- **Reconnect backoff timing**: linear vs exponential vs custom — audit Plan 01 decides.
- **Memory toast duration**: 3-5 seconds, dismissible — agent's choice within that range.
- **Stop button confirmation**: instant stop vs confirmation modal — instant recommended (matches grid-cli `session kill` behavior).
- **Layout shift prevention**: SessionBar position fixed bottom-right may need z-index adjustment — handled in audit Plan 01.

## Out of scope (deferred)

- `web-platform/` multi-tenant UI 实战化 → future milestone (Phase 3.7.x or v3.8)
- `grid-desktop` Tauri app feature completion → future milestone
- New pages (e.g. dedicated "Agent Live Monitor" page) — out of scope; SessionBar promotion is the affordance path
- Server-side changes to WS event format beyond adding `seq` field — additive only, no breaking changes
- WebSocket compression / binary protocol — premature optimization

## Acceptance Standard

1. `npm run test` (vitest) PASS — ≥ 9 prior tests + new SessionBar tests
2. `npx playwright test` PASS — E1 (Stop), E2 (memory toast), E3 (WS reconnect) all green
3. Manual walkthrough (`docs/status/WEB_PRODUCTION_USABILITY_<date>.md`) shows S3 hook governance scenario completing end-to-end through `web/` without devtools intervention
4. SessionBar Stop button: visible during `running`, triggers `POST /sessions/:id/kill`, UI updates to `stopped` within 2s
5. SessionBar Resume button: visible after stop, triggers `POST /sessions/:id/resume`, UI updates to `running`
6. Memory write toast: appears within 1s of agent's `memory_added` event, dismissible, also visible on Memory page
7. WS auto-reconnect: when server killed mid-session, client reconnects with same session_id within 30s
8. Sequence number (`seq` field): present in all WS messages, UI debug mode logs gaps
9. `docs/audit/3.7.2-GAP-AUDIT.md` exists with 8×5 matrix + REQ-WEB backlog
10. `docs/cli/scenarios/S7-web-dashboard.md` exists with prerequisites + walkthrough steps
11. ADR-V2-023 P1 dual-leg: `web/` is `B only` per CLAUDE.md; no changes cross into leg A
12. `cargo check --workspace` clean (web/ changes should not affect Rust crates; verify)
