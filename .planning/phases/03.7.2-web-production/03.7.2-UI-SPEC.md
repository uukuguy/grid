# Phase 3.7.2 — web/ UI Design Contract

> Visual and interaction contract for the Grid single-user workbench UI (leg B per ADR-V2-024). Generated by `gsd-ui-researcher`, verified by `gsd-ui-checker`. Drives Plan 3.7.2-02 implementation.

**Phase goal (from ROADMAP):** Close the gap between Activation-9.0 and 实战不可用 for `web/`. Build a dashboard a non-developer can open, observe a running agent, see its tool calls, see its memory writes, and stop/resume it without devtools intervention.

**User priority (D-09, 2026-07-20):** modern, professional, refined visual design + interaction design quality. Phase 3.7.2 elevates `web/` from "works" to "delightful."

## 1. Design System

| Property | Value |
|----------|-------|
| Tool | **none (custom Tailwind 4 + cva stack)** |
| Preset | not applicable (no shadcn CLI) |
| Component library | none (project uses Tailwind 4 utilities + `cn()` helper) |
| Icon library | `lucide-react` 0.469 (existing) |
| Font | system font stack (existing: `-apple-system, BlinkMacSystemFont, "Segoe UI", Roboto, sans-serif`) |
| CSS framework | Tailwind CSS 4.0 (theme tokens in `web/src/globals.css`) |
| State primitive | Jotai 2.16 (existing; no Redux/Zustand) |
| Markdown | `react-markdown` 10 + `remark-gfm` + `rehype-highlight` (existing) |
| Existing utilities | `class-variance-authority`, `clsx`, `tailwind-merge` (already in `package.json`) |

**Rationale — why not introduce shadcn:** `web/` is already a hand-rolled Tailwind 4 + cva dashboard with 8 tabs, 9 vitest tests, and a working design-token system defined in `globals.css`. Introducing shadcn now would (a) duplicate token systems, (b) require rewriting 8 page components, (c) conflict with the existing `class-variance-authority` direct usage. The `cn()` helper at `web/src/lib/utils.ts:4` already follows the shadcn convention. We extend the existing system rather than replace it.

**Rationale — why not introduce a font:** The system font stack is fast (no FOUT), OS-native (looks at home on macOS, Windows, Linux), and matches the `grid-cli` TUI's monospace-first feel. Adding a webfont (Inter, Geist, etc.) would add ~80KB + ~200ms FCP for marginal gain. The current system font achieves the "professional, refined" bar.

## 4. Typography

**One type family, four sizes, two weights, two line-heights** (intentionally minimal — matches the "professional, refined" brief without typography noise).

| Role | Size | Weight | Line Height | Tailwind | Usage |
|------|------|--------|-------------|----------|-------|
| Body | 14px | 400 (regular) | 1.5 | `text-sm font-normal leading-normal` | Default body text, list items, table cells |
| Label | 12px | 500 (medium) | 1.4 | `text-xs font-medium leading-tight` | Form labels, badges, captions, tab labels, button text |
| Heading | 18px | 600 (semibold) | 1.25 | `text-lg font-semibold leading-tight` | Page titles, modal titles, section headers |
| Display | 24px | 600 (semibold) | 1.2 | `text-2xl font-semibold leading-tight` | Hero numbers, dashboard summaries (rare) |

**Code/monospace** (already used in `SessionBar.tsx:153`, `Tasks.tsx:184`):
- Font: `ui-monospace, SFMono-Regular, "SF Mono", Menlo, Consolas, monospace`
- Size: 12px (`text-xs`)
- Weight: 400 (regular)
- Line-height: 1.4 (`leading-tight`)
- Tailwind: `font-mono text-xs`

**Rationale — why 14/12/18/24:** The current code already uses `text-sm` (14px) for body, `text-xs` (12px) for labels/badges, `text-lg` (18px) for page titles. The only NEW size is `text-2xl` (24px) for the SessionBar's "session #N" status when the dashboard is at-rest. This is **not adding typography** — it's formalizing what the code already does.

**Rationale — why only two weights:** 400 + 600 give us 2 visual tiers (body, emphasis). Adding 500 creates a "middle" that's hard to use meaningfully; adding 700 violates the "refined" aesthetic. This matches modern 2026 dashboard conventions (Linear, Vercel, Stripe, Notion all use 2 weights).

## 7. Motion / Transition

**Three durations, two easings.** Modern 2026 dashboard convention: short, snappy, never bouncy.

| Token | Duration | Easing | Tailwind | Usage |
|-------|----------|--------|----------|-------|
| Fast | 150ms | ease-out | `transition-colors duration-150` | Color changes (hover, active) — by far the most common |
| Normal | 200ms | ease-out | `transition-all duration-200` | Slide-in for toasts, button press feedback |
| Slow | 400ms | ease-in-out | `transition-all duration-400` | Page transitions (rare; mostly none in this phase) |

**Easing rules:**
- `ease-out` for elements entering (toast slide-in, dropdown open)
- `ease-in` for elements leaving (toast dismiss — `transition-opacity duration-200 ease-in`)
- No `cubic-bezier(0.68, -0.55, 0.27, 1.55)` bouncy curves — they look toy-like.

**Live indicator pulse (the only continuous animation):**
- `animate-pulse` (Tailwind built-in, 2s cycle) on the SessionBar live dot
- This is the only continuous animation in the dashboard. Everything else transitions on user action.

**Reduce motion respect:** Wrap all motion in `@media (prefers-reduced-motion: reduce)` overrides. Use Tailwind's `motion-reduce:` prefix. Example: `motion-reduce:animate-none motion-reduce:transition-none`. Add to `globals.css` after Plan 02 ships the new components.

**Existing transitions in code (do not regress):**
- `SessionBar.tsx:148,157,173` — `transition-colors` on session pills, close icons, new-session button
- `ConnectionStatus.tsx:32` — implicit via `animate-pulse`
- `Toast.tsx:58` — `transition-opacity` on close button hover

**Position:** Already rendered inside `AppLayout.tsx:16` when `activeTab === "chat"`. **No change to position** — D-02's "global controller" framing is about *function*, not chrome. The SessionBar remains a chat-tab component per existing pattern.

**Where Stop/Resume live (D-02 specifies a global controller, but the current code only renders SessionBar on the chat tab):**

**Decision:** Stop/Resume buttons render inside the existing SessionBar. They are visible whenever there is an active session (regardless of which tab is active) — accomplished by lifting the Stop/Resume JSX out of the SessionBar's chat-only branch and into a new `SessionControls.tsx` component that is mounted in `AppLayout.tsx` at all times. The session pills + live indicator stay inside SessionBar (chat-only, as today).

**Layout split:**
- `SessionBar` (chat-only) — session pills, new-session button, ConnectionStatus. UNCHANGED structure.
- `SessionControls` (always-visible) — Stop button + Resume button + live indicator dot. NEW component, mounted in `AppLayout.tsx` next to the existing children.

**Live indicator (D-02 affordance, pulsing dot during streaming):**

| State | Color | Animation | Size |
|-------|-------|-----------|------|
| Running (`isStreamingAtom === true`) | `bg-primary` (#3b82f6) | `animate-pulse` (2s cycle) | `h-2 w-2 rounded-full` (8px) |
| Idle (no active session OR session not streaming) | `bg-emerald-500` (#22c55e) | none | `h-2 w-2 rounded-full` (8px) |
| Disconnected | `bg-red-500` (#ef4444) | none | `h-2 w-2 rounded-full` (8px) |

**Rationale — three states for one dot:** A non-developer watching the dashboard needs to know "is the agent working right now?" (pulsing blue = yes), "is the agent ready to take input?" (steady green = yes, idle), "is the connection broken?" (steady red = no, call for help). Three states map to three concrete actions: wait, type, refresh. This is the heart of D-02's "live indicator."

**Stop button spec (D-02, instant stop per D-08):**

| Property | Value |
|----------|-------|
| Visibility | Visible only when `sessionStatusAtom === "running"` (i.e. when `isStreamingAtom === true` OR a `tool_execution` event has `status === "running"`) |
| Default state | `bg-secondary text-foreground border border-border rounded-md px-3 py-1.5 text-xs font-medium` |
| Hover state | `bg-destructive/10 text-destructive border-destructive/30` (subtle red tint, not full red) |
| Active/pressed | `bg-destructive/20 text-destructive scale-95` (brief 150ms scale-down for press feedback) |
| Disabled (while in-flight) | `opacity-50 cursor-wait` — locked while the `POST /sessions/:id/kill` request is in flight |
| Focus | `ring-2 ring-primary ring-offset-2 ring-offset-background` (universal focus ring) |
| Icon | `Square` (lucide-react), `h-3.5 w-3.5`, `fill-current` (filled square, not outline) |
| Label | "Stop" (verbatim — single word, verb) |
| Touch target | 44×44px on mobile (wraps the visible button in a `min-h-11 min-w-11` flex container) |
| Backend action | `POST /api/v1/sessions/:id/kill` (mirrors `grid session kill`) |
| Optimistic UI | On click: button enters `disabled` state IMMEDIATELY (no spinner — D-08 says instant). When `isStreamingAtom` flips to false OR a `done`/`error` event arrives: button hides, Resume button shows. |
| Failure handling | On HTTP 4xx/5xx: revert `disabled` state, show error toast "Failed to stop session. Try again." Do NOT retry automatically. |
| No confirmation modal | Per D-08: instant stop. The visual feedback is: the button disables, the live indicator stops pulsing within 2s, and the agent stops. No `confirm()`. |

**Resume button spec (D-02, complementary to Stop):**

| Property | Value |
|----------|-------|
| Visibility | Visible only when `sessionStatusAtom === "paused"` or `"stopped"`. NOT visible during `"running"`. |
| Default state | `bg-primary text-primary-foreground rounded-md px-3 py-1.5 text-xs font-medium` |
| Hover state | `bg-primary/90` (slightly darker) |
| Active/pressed | `bg-primary/80 scale-95` (brief scale-down) |
| Disabled (while in-flight) | `opacity-50 cursor-wait` |
| Focus | `ring-2 ring-primary ring-offset-2 ring-offset-background` |
| Icon | `Play` (lucide-react), `h-3.5 w-3.5`, `fill-current` |
| Label | "Resume" |
| Touch target | 44×44px on mobile |
| Backend action | `POST /api/v1/sessions/:id/resume` |
| Optimistic UI | On click: button disables, `isStreamingAtom` flips to `true` (assumed), live indicator starts pulsing. When `text_delta` event arrives: confirmed. When `error` event arrives: revert + error toast. |

**Why Stop is gray-default and Resume is blue-primary:** Per D-08, Stop is **instant** (no confirmation), so it must be a normal control, not a danger button. Resume is the **constructive action** — the thing the user wants to do after stopping. So Resume is the one that gets the primary accent. This is consistent with iOS, macOS, and modern web app conventions (e.g. Spotify: pause is gray, play is green/primary).

**The `sessionStatusAtom` derivation (new):**

Add to `web/src/atoms/session.ts`:

```ts
// Derived from isStreamingAtom + tool execution state
// "running" | "paused" | "stopped" | "idle"
export const sessionStatusAtom = atom<'running' | 'paused' | 'stopped' | 'idle'>((get) => {
  const isStreaming = get(isStreamingAtom);
  const hasRunningTool = get(toolExecutionsAtom).some((t) => t.status === 'running');
  if (isStreaming || hasRunningTool) return 'running';
  // "paused" vs "stopped" needs a separate flag (set by Stop click, cleared by Resume click)
  // For Phase 3.7.2, default to "stopped" (user explicitly stopped)
  const stopped = get(stoppedByUserAtom);
  return stopped ? 'stopped' : 'idle';
});

export const stoppedByUserAtom = atom<boolean>(false);
```

`stoppedByUserAtom` flips to `true` when the Stop button is clicked, flips to `false` when the Resume button is clicked OR a new session starts.

**Copy for the Stop button's aria-label:**

```html
aria-label="Stop session ${truncateId(sessionId)}"
```

For Resume:
```html
aria-label="Resume session ${truncateId(sessionId)}"
```

### 9.2 Toast — memory_write variant (D-02)

**Existing Toast.tsx already handles 4 types:** `success` | `error` | `warning` | `info`. Add a 5th: `memory`.

**New `memory` type — what it looks like:**

| Property | Value |
|----------|-------|
| Icon | `Database` (lucide-react), `h-5 w-5`, `text-cyan-400` |
| Border | `border-cyan-600` |
| Background | `bg-cyan-950/80` (matches existing pattern at `Toast.tsx:20-23`) |
| Text | `text-cyan-200` (matches existing pattern) |
| Title | "Memory written" (verbatim) |
| Message | "Stored: {first 60 chars of memory content}…" (truncate with ellipsis) |
| Default duration | 4000ms (longer than 3s — D-09 says "delightful", give time to read; but not 5s+ to avoid clutter) |
| Position | `right-4 top-4` (existing, `Toast.tsx:72`); multiple memory toasts stack vertically |
| aria-live | `polite` (existing `role="alert"` at `Toast.tsx:48` — keep) |
| Action button | None (memory toasts are informational; clicking the Memory tab is the next step) |

**Why cyan, not primary blue:** Cyan is distinct from the primary `blue-500` so the user visually knows "this is not a regular info message — this is a memory write." Cyan also pairs well with the existing emerald/yellow/red semantic palette without overlap.

**Why no action button:** Adding a "View in Memory" link inside the toast adds visual weight and competes with the toast's job (inform briefly, then dismiss). The Memory page is one tab click away; the toast's job is to say "this just happened" and let the user decide.

**Memory toast is the ONLY variant added in this phase.** The existing 4 variants (`success`/`error`/`warning`/`info`) keep their styling. Decision: don't reuse `info` for memory — they're conceptually different (info is generic; memory is a specific event type with a specific affordance).

**Trigger:** `ws/events.ts` handles the `memory_added` server event (already plumbed, see `events.ts:184-191` for `memory_flushed` which is similar). Plan 02 wires `memory_added` to `addToastAtom` with `type: "memory"`.

**Snippet (target in `web/src/ws/events.ts`):**

```ts

    type: "memory",
```

### 9.4 Memory page — live-update affordance (D-02)

**Current state** (`Memory.tsx`): manual fetch on mount, session-change, and filter-change. **No real-time updates.**

**Affordance 1: New-memory highlight pulse**

| Default row | `bg-transparent` (or `bg-card`) | continuous |

| Fading (3-6s) | transition to `bg-transparent` over 1500ms | 1500ms |
| Settled | `bg-transparent` | continuous |

**Implementation note:** A new atom `recentlyAddedMemoryIdsAtom: Set<string>` tracks IDs. The `Memory.tsx` persistent view maps each row, and if its ID is in the set, applies the highlight class. The atom auto-clears the ID after 4.5s (3s highlight + 1.5s fade buffer).

**Affordance 2: Subtle "Live" badge in the page header**

```html

```

The badge is always present (not just when new memories arrive) — it tells the user "this page auto-updates; you don't need to refresh." This is the **proactive disclosure** of the live-update behavior.

**Affordance 3 (optional, only if Affordance 1 is too noisy): an unobtrusive toast on memory_added**

Already covered by §9.2 (the memory toast variant). If Affordance 1 (the highlight pulse) is implemented, the toast becomes optional. **Default: both fire.** Users can dismiss the toast; the pulse stays. This is the "delightful" layering D-09 asks for.

**Why not auto-scroll to the new memory:** A non-developer observer should not have their place in the page yanked away by an auto-scroll. The highlight + Live badge are *informative* ("something new happened at this row") but *non-interrupting* ("your scroll position is preserved"). The user can click the row to focus it.

## 10. Interaction Patterns

### 10.1 Button states — universal

| State | Tailwind | Applies to |
|-------|----------|------------|
| **Default** | (component-specific) | All buttons |
| **Hover** | Slight color shift (5-10% opacity change) | All buttons |
| **Active (pressed)** | `scale-95` + color intensifies | All buttons (D-09: "delightful micro-feedback") |
| **Focus** | `ring-2 ring-primary ring-offset-2 ring-offset-background` | All interactive elements (WCAG 2.4.7) |
| **Disabled** | `opacity-50 cursor-not-allowed` (NOT `pointer-events-none` — keep element focusable for screen reader announcement) | All buttons |
| **Loading/in-flight** | `opacity-50 cursor-wait` + optionally `animate-pulse` on the icon | Action buttons only |

**Loading spinner policy:** **No spinners.** The Task page Submit button uses `"Submitting..."` text instead of a spinner (`Tasks.tsx:152`). This pattern is the project convention. D-09 prefers text feedback over visual noise.

### 10.2 Keyboard shortcuts

| Shortcut | Action | Scope | Notes |
|----------|--------|-------|-------|
| `Cmd+.` (macOS) / `Ctrl+.` (others) | Stop current session | When SessionControls is mounted | Cross-platform; matches VS Code "stop running task" convention |
| `Esc` | Cancel current focus / close any open modal | Global | Already implicit in browser default; explicit handling for future modals |
| `Enter` (in Chat input) | Send message | Chat input only | Existing behavior (`Tasks.tsx:143`) |
| `Tab` / `Shift+Tab` | Move focus through interactive elements | Global | Native; verify focus order is logical (left-to-right, top-to-bottom) |
| `Cmd+K` / `Ctrl+K` | (future) Quick command palette | Not in this phase | Reserved |

**Stop shortcut implementation note:** the `Cmd+.` shortcut lives on the `SessionControls` component. It calls the same `handleStop()` function the button calls. It's only active when `sessionStatusAtom === "running"`. The shortcut is **registered globally** (on `window`) so the user can hit it from any tab, but the visual focus indicator (the Stop button's focus ring) only appears when the user navigates to it via Tab. D-09 loves keyboard shortcuts because non-developers who type 100wpm use them instinctively.

**Disclosure:** A small tooltip or help-text should mention the `Cmd+.` shortcut somewhere visible (e.g. on the Stop button's `title` attribute, or a one-line hint at the bottom of SessionBar). Tooltip copy: `Stop session (⌘.)` on macOS, `Stop session (Ctrl+.)` elsewhere. Use `navigator.platform` (or `navigator.userAgent`) to detect macOS; this is a one-time check at component mount.

### 10.3 Loading / success / error states per affordance

### 10.5 Confirmation vs instant action

| Action | Confirmation | Reason |
|--------|--------------|--------|
| **Stop session** | None (instant) | Per D-08: "instant stop recommended (matches grid-cli `session kill` behavior)" |
| **Resume session** | None (instant) | Constructive; user wants to resume, no undo needed |
| **Delete task** | `window.confirm()` modal | Destructive + non-reversible (current `Tasks.tsx:96` behavior — keep) |
| **Close session** | None (instant) | Low-stakes; if it's the last session, show warning toast (current `SessionBar.tsx:97-105` behavior — keep) |
| **Clear memory** | `window.confirm()` modal | Future; not in this phase |

**Stop button visual feedback that confirms the action landed (D-09 asks for this explicitly):**

Within 500ms of the Stop click:
1. The button enters `disabled` state (instant).
2. The live indicator dot transitions from `bg-primary animate-pulse` to `bg-emerald-500` (steady green = "stopped, ready") — this is the visual confirmation.
3. The Resume button fades in over 200ms (replaces the Stop button).

The user knows the action landed because: (a) the button they clicked visibly changed state, (b) the pulsing dot stopped, (c) a new button appeared. Three signals, no toast, no modal. This is the "delightful" 2026 dashboard pattern.

### 10.6 Error states (universal)

| Source | UI pattern |
|--------|------------|
| HTTP 4xx from any backend call | `addToastAtom` with `type: "error"`, `title: "Action failed"`, `message: <response message>` |
| HTTP 5xx | `addToastAtom` with `type: "error"`, `title: "Server error"`, `message: <response message>`, no auto-retry |
| Network error (fetch rejects) | `addToastAtom` with `type: "error"`, `title: "Connection error"`, `message: "Check your network and try again"` |
| WebSocket disconnect | `ConnectionStatus` dot goes red; no toast (avoid toast spam during reconnect) |
| WebSocket reconnect failure (5 attempts) | One toast: `type: "warning"`, `title: "Disconnected"`, `message: "Refresh the page to reconnect."` |
| Optimistic UI rollback (e.g. Stop failed) | Revert button state + error toast |

## 12. Accessibility (WCAG 2.1 AA)

### 12.1 Color contrast

| Foreground | Background | Ratio | Pass? |
|------------|------------|-------|-------|
| `#fafafa` (text-foreground) | `#09090b` (bg-background) | 18.8:1 | AAA (text 7:1 minimum) |
| `#a1a1aa` (text-muted-foreground) | `#09090b` (bg-background) | 9.4:1 | AAA |
| `#a1a1aa` (text-muted-foreground) | `#27272a` (bg-secondary) | 5.2:1 | AA (text 4.5:1 minimum) |
| `#3b82f6` (text-primary) | `#09090b` (bg-background) | 5.7:1 | AA |
| `#3b82f6` (text-primary) | `#27272a` (bg-secondary) | 3.2:1 | AA Large only (3:1 minimum) — body text on secondary needs a different color |
| `#fafafa` (text-primary-foreground) | `#3b82f6` (bg-primary) | 5.7:1 | AA |
| `#ef4444` (text-destructive) | `#09090b` (bg-background) | 5.0:1 | AA |
| `#22c55e` (text-emerald-500) | `#09090b` (bg-background) | 5.5:1 | AA |
| `#22c55e` (text-emerald-400 — for text on dark) | `#09090b` (bg-background) | 8.4:1 | AAA |
| `#eab308` (text-yellow-500) | `#09090b` (bg-background) | 8.0:1 | AAA |
| `#22d3ee` (text-cyan-400) | `#09090b` (bg-background) | 10.2:1 | AAA |

**Body text on `bg-secondary`:** use `text-foreground` (`#fafafa`) — 16.0:1, AAA. Do not use `text-muted-foreground` on `bg-secondary` for body text; reserve it for captions.

**Caption text on `bg-secondary`:** `text-muted-foreground` (`#a1a1aa`) — 5.2:1, AA. Acceptable for captions (which are not body text per WCAG).

**Primary text on `bg-primary`:** use `text-primary-foreground` (`#fafafa`). Never put colored text on `bg-primary`.

**Decision — no color-only signaling:** Status (success/warning/error) is always paired with an icon AND text label. The color is reinforcement, not the only signal. (E.g. toast: icon + title + message + color, all four together.)

## 16. Design Rationale (the "Why" for each major decision)

### 16.1 Why SessionBar is the affordance, not a new "Agent Monitor" page

D-02's locked decision. The user could have asked for a new page; instead, we promote an existing component. **Why this is right:**
- SessionBar is already mounted (in the chat tab) and already reads `isStreamingAtom` and `sessionsAtom` — no new global state setup.
- Adding a new page forces the user to navigate to see status; the affordance should be omnipresent, not page-dependent.
- The session *is* the live state — the pill row is the dashboard, conceptually.
- The cost of promoting is small (one component split, one new atom). The cost of a new page is large (new route, new layout, new state, new tests).

### 16.2 Why Stop is gray-default, not red-default

Per D-08: instant stop, no confirmation. **Why this works:**
- A red Stop button implies "are you sure?" — we explicitly do NOT want that implication.
- A gray Stop button is a normal control. The user can find it without being warned off.
- On hover, the button flashes red. This tells the user "this is destructive" right at the moment of action, not before.
- Modern apps follow this pattern: Spotify (pause is gray, stop is hidden in a menu), YouTube (stop is gray, delete is red), VS Code (stop is gray square, kill terminal is red).

### 16.3 Why cyan for memory toasts, not another shade of primary

- Primary blue is reserved for "primary action in progress" (running state). Reusing it for memory would make the two signals collide.
- Cyan is adjacent to blue on the color wheel (similar hue, different saturation/brightness) — it reads as "related to data, not a primary action."
- Cyan pairs naturally with the dark background (10.2:1 contrast, AAA).
- Cyan is not used anywhere else in the dashboard (green = success, yellow = warn, red = error, blue = primary action, purple = json keys in debug). It's the only color "free" for memory semantics.

### 16.4 Why no auto-scroll on memory add

- Auto-scroll yanks the user away from where they were reading.
- The new-memory highlight pulse + Live badge tell the user "something new happened at this row" without moving them.
- The user can choose to scroll by clicking the row.
- This matches Linear's notification pattern (badge appears, no auto-jump).

### 16.5 Why skeletons, not spinners

- Skeletons preserve layout (no content shift when data loads).
- Spinners say "wait, unknown time" — skeletons say "here's the shape, just a moment."
- Skeletons are easier on the eye (subtle pulse, not a spinning element).
- 2026 dashboard convention: Linear, Vercel, Stripe, Notion all use skeletons.

### 16.6 Why `Cmd+.` for Stop

- `Cmd+.` is the universal "stop" shortcut in macOS (e.g. terminal `Ctrl+C` is also `Cmd+.` in some contexts; VS Code uses it for "stop running task").
- `Ctrl+.` is the cross-platform equivalent.
- The shortcut is **discoverable** via the Stop button's `title` tooltip.
- This shortcut does NOT conflict with browser defaults (`Cmd+.` does nothing in Chrome/Safari by default).
- The shortcut is **global** (registered on `window`), so it works from any tab.

### 16.7 Why only two font weights

- 400 (regular) for body = readable.
- 600 (semibold) for emphasis = noticeable but not heavy.
- 500 (medium) would create a "middle" that's hard to use meaningfully.
- 700 (bold) would feel heavy, violate the "refined" aesthetic.
- This matches Linear, Vercel, Stripe.

### 16.8 Why no webfont

- System fonts are fast (no FOUT, no FOIT).
- System fonts are OS-native (feels at home on every platform).
- The Tailwind default font stack uses SF Pro on macOS, Segoe UI on Windows, Roboto on Android — all excellent.
- Adding Inter or Geist would add ~80KB + ~200ms FCP for marginal aesthetic gain.
- The "professional, refined" bar is met by good typography **hierarchy** (size + weight + spacing), not by font choice.

### 16.9 Why dark mode only (no light mode toggle)

- The existing `globals.css` declares only dark-mode tokens.
- All 8 pages are dark-mode-only (no `[data-theme="light"]` selectors).
- Adding a light mode would require defining a parallel set of 16+ tokens, retesting every page for contrast, and adding a theme toggle UI.
- Dark mode is the "professional, refined" default for 2026 dashboards (Linear, Vercel, Stripe Dashboard, Raycast, Arc Browser, etc.).
- If a future phase needs light mode, the token system in `globals.css:4-22` is already structured to accept it (CSS custom properties make it trivial to add `[data-theme="light"]` overrides). Not in this phase.

### 16.10 Why 8px spacing is the default

- 8-point grid is the industry standard (Material, iOS, Fluent, Carbon all use it).
- 8 is divisible by 2, 4, 8 — all the common screen pixel ratios align cleanly.
- The existing `SessionBar.tsx:141` uses `h-9` (36px = 4×8+4) and `p-2.5` (10px = 8+2) — already on the grid.
- The existing `Tasks.tsx:140` uses `px-3 py-2` (12px, 8px) — already on the grid.
