## Common Pitfalls

### Pitfall 1: STATE / LEDGER text drift vs ADR frontmatter

**What goes wrong:** STATE.md, DEFERRED_LEDGER.md, or even prior CONTEXT.md summaries describe an ADR's status from memory or from a stale ledger entry. The ADR's actual frontmatter has moved on.
**Why it happens:** ADRs flip status outside the dev-phase-manager session that wrote the STATE/LEDGER row. The text behind summarizes; the frontmatter is the SoT.
**How to avoid:** Before any ADR-related task (re-edit, status check, supersede), `head -25 docs/design/EAASP/adrs/<id>.md` to read the live frontmatter. MEMORY.md captures this as an always-on rule.
**Warning signs:** "ADR ought to be Proposed but lint says Accepted" — that means the lint is right.
**Phase 5.5 application:** Already applied — RESEARCH §2.4 verified ADR-V2-019 frontmatter live (it shows trace filled per 5.4-02-10). ADR-V2-026 / V2-027 / V2-028 confirmed Accepted (consumed in Phase 5.3 / 5.4 respectively).

### Pitfall 2: PLAN spec drift contaminating subsequent phases

**What goes wrong:** A PLAN authored two phases ago hard-codes a value that's wrong (e.g. "EXIT_SESSION = 71") and a downstream phase trusts it without re-verifying against the catalog.
**Why it happens:** PLANs are point-in-time artifacts; the code may drift faster than the PLAN.
**How to avoid:** Treat PLAN constants as hypotheses. When CONTEXT.md flags "PLAN said X but STATE table says Y", lock the value by reading the catalog directly.
**Phase 5.5 application:** §2.1 resolved: PLAN T-01.16 said `EXIT_SESSION = 71`. Live catalog at `crates/grid-cli/src/error.rs:28` says `SessionNotFound = 4`. **Catalog wins.** Plan must lock the constant by referencing `ExitCode::SessionNotFound` (not by inventing a numeric literal).

### Pitfall 3: "Stale issue" carrying forward via copy-paste

**What goes wrong:** A deferred-items file lists an issue that has been quietly resolved by an unrelated commit; subsequent CONTEXT.md inherits the entry verbatim.
**Why it happens:** Deferred items are append-only by convention; cleanup is opportunistic.
**How to avoid:** Re-validate every deferred item at phase entry. Run the failing command (if applicable) and re-check the symptom.
**Phase 5.5 application:** §2.2 resolved: grid-cli `mod output` E0583 is already fixed. `cargo build -p grid-cli` and `cargo build -p grid-cli --features studio` both succeed clean as of 2026-05-22. The "fix" task collapses to a verification-and-close task.

### Pitfall 4: F3 trace WARN baseline arithmetic drift

**What goes wrong:** ROADMAP / earlier CONTEXT cited "29 missing items" but the live baseline is **33 WARN across 9 ADRs**. Outdated baselines push down the target ("≤ 5 unjustified") and create false success claims.
**Why it happens:** F3 WARN count varies with both ADR additions (V2-027 + V2-028 added 4 WARN since the 29 number was cited) and the set of CI workflows that exist.
**How to avoid:** Run `python3 .adr-plugin/scripts/adr_lint.py 2>&1 | grep -c '^\[F3 WARN\]'` at phase entry to lock the actual baseline. SC#3 wording must be revised per D-04 to "0 unjustified + 13 explicit-strategic" (NOT "≤5 unjustified").
**Phase 5.5 application:** §2.4 captures the live distribution. Plan 02 SC#3 revision is required.

### Pitfall 5: Confusing "advisory" with "non-blocking" in autonomous executor mode

**What goes wrong:** WATCH-06 / NEW-E2 is labeled "advisory" — but D-04/D-05 actively require frontmatter edits across 9 files. If a plan task interprets "advisory" as "skip", the F3 sweep doesn't happen.
**How to avoid:** Plan 01 must include 5 contract-ADR trace-fix tasks + 4 strategy-ADR rationale-comment tasks (or a single batched task with 9 sub-edits). Acceptance criteria: F3 WARN drops to ≤ 13 strategic-tagged.
