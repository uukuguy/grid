## Implementation Decisions

### GA1 — ADR-V2-029 scope / depth

- **D-01:** **ADR depth = crate-level enforcement rules.** Engine side enumerates `crates/grid-*` + `tools/eaasp-l[2-4]-*` + `proto/eaasp/runtime/v2/` (as already drafted in ROADMAP SC#1). Data/integration side enumerates categories: customer data ingestion adapters / SSO contracts / third-party API gateway / WORM 存储 / 信创 LLM 适配 / hook-out 接入面. **No code skeleton, no trait signatures, no proto declarations** — those are V2-030 / V2-031 (deferred).
- **D-02:** **ADR type = `strategy`, `enforcement.level: strategic`.** Same shape as V2-017 / V2-023 / V2-024 / V2-025. Allows `trace=[]` with rationale since ADR-only / no concrete code to trace. **Do NOT use `type: contract`** — contract type would force backfilling traces against crate-boundary enforcement that hasn't been implemented yet (that's V2-030's job).
- **D-03:** **Future ADR placeholders named explicitly in V2-029 §References:** `V2-030` (code-level boundary enforcement — Cargo dep-checks, F4 module-overlap reconcile, etc.) and `V2-031` (Rust trait + gRPC service skeleton for data/integration hook-out sites).
