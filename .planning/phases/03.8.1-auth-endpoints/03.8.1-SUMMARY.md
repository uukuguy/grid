## What landed

- **AuthConfig.mint_jwt** symmetric to **validate_jwt** (03.8.0) — both use HS256 from `grid-engine::secret::vault`'s deps.
- **UserStore** (engine-side, per ADR-V2-023 P1): in-memory HashMap, seeded from `GRID_USERS_JSON`, Argon2id-hashed at load. 5 inline unit tests cover round-trip + reject paths.
- **TokenBlacklist** (engine-side): in-memory `Mutex<HashMap<jti, exp>>` with lazy GC. 3 inline unit tests.
- **JwtClaims.tenant_id** (03.8.0) + **JwtClaims.jti** (03.8.1) — both required (D-05 cascading contract).
- **AuditEvent** gains `tenant_id: Option<String>` + `role: Option<String>`; **AuditRecord** mirrors; **migration v14** adds the columns (idempotent via `user_version` gate).
- **Audit middleware** (grid-server) reads `Extension<JwtClaims>` and tags every authenticated request's audit row.
- **3 HTTP handlers** (`crates/grid-server/src/api/auth.rs`): `login_handler`, `refresh_handler`, `logout_handler` — wired at `/api/v1/auth/{login,refresh,logout}`.
- **AppState** extends with `users`, `token_blacklist`, `token_ttl_secs`.
- **6 hermetic tests** in `crates/grid-server/tests/multi_user_auth_endpoints.rs`.

## Known limitations called out in code

- Login timing-side-channel (`UserStore::verify_credentials` short-circuits on unknown email). Documented in the module doc-comment; mitigation deferred to v3.9+ login rate-limiting.
- TokenBlacklist is in-memory only. Multi-instance deployments need a shared backend (Redis, DB) — v3.9+.
- No login rate limiting → brute-force-friendly until v3.9+.
- `AppState::new` panics on bad `GRID_USERS_JSON` parse in `AuthMode::Full` (per D-03). In other modes it parses fail-soft and falls back to `UserStore::empty()` (login disabled).

## Notes for 03.8.2

- The `Extension<JwtClaims>` middleware path is now established; the `requires(Action)` RBAC middleware in 03.8.2 must run *after* the AuthMode::Full middleware so the claims extension is populated.
- The audit `tenant_id`/`role` columns are populated in `AuthMode::Full` only. 03.8.2 needs no audit-side changes; just the handler-level scoping check (`resource_id` belongs to `claims.tenant_id`?).
- Pre-flight kill-switch tests for D-05 exist (hand-crafted JWTs missing `tenant_id`/`jti` → 401). 03.8.2's cross-tenant tests should reuse that harness.
