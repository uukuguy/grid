# Phase 03.8.2 — RBAC route enforcement + TenantContext::for_multi_user + Cross-tenant isolation

## Goal (1 sentence)

Wire `Extension<JwtClaims>` (set by 03.8.0's Full-mode middleware) into a per-request authorization context, then enforce `(tenant_id == claims.tenant_id)` + role-membership checks at the handler / data-access layer so cross-tenant reads return 403 and high-risk actions (user-management, audit) require `Admin`/`Owner`.

## Why this scope is large but tractable

- The four 03.8.1 commits already established: `Extension<JwtClaims>` is populated whenever `auth.mode = full`.
- The `Role × Action` matrix already exists at `grid_engine::auth::roles::Role::can(Action)` — 8 actions × 4 roles already covered (verified in [[project_v3_8_1_summary]] scope).
- The `require_action_middleware` skeleton already lives at `crates/grid-server/src/middleware/auth.rs:206` and maps each `Action` to a `Permission`. It just isn't wired to handlers.
- `TenantContext` already has `for_single_user` and `can(action)` — `for_multi_user` is the only missing constructor; `can` already iterates `roles` so adding multi-role is free.

## Locked decisions

| # | Decision | Source |
|---|----------|--------|
| D-01 | `TENANT-01` contract: every Full-mode request reads `JwtClaims` → resolves `TenantContext { tenant_id, user_id, role }`. No new constructor needed; the existing `TenantContext` already accepts `Vec<Role>` — so `for_multi_user` returns `roles: vec![record.role]` (single role per user in v3.8.2; multi-role provisioning is v3.9+). | scope discipline |
| D-02 | `RBAC-01` via the existing `require_action_middleware(Action)` — extend it to also check `claims` (already in extension) against `Role::can(action)`. For `AuthMode::None/ApiKey` paths the existing path stays unchanged; v3.8.2 only adds the JWT-aware branch. | existing pattern (auth_middleware_with_role) |
| D-03 | `RBAC-02` (POST /api/v1/users requires ManageUsers): in v3.8.2 there is no `POST /users` endpoint (user-management is v3.9+); the rule is enforced by the auditor test that checks every route's RBAC metadata, not by a missing endpoint. The regression test for RBAC-02 hits a /session-by-id endpoint and verifies `Admin` JWT gets 200, `Viewer` JWT gets 403. | scope discipline |
| D-04 | `RBAC-04` (Owner always succeeds): pre-existing — `Role::Owner`'s arm in `Role::can` returns true for every action. Verification: hermetic test asserts `Owner` JWT calling any RBAC-protected endpoint always 200. | preserve |
| D-05 | `TENANT-03` (cross-tenant data access → 403): enforced at the SessionStore read helper level — `SessionStore::get(id)` becomes `SessionStore::get_for_tenant(id, tenant_id)` which returns `Result<Session, TenantMismatch>`; handlers in multi-user mode always call this. Single-user mode (default `for_single_user` + single tenant) keeps the existing un-scoped path. | least-invasive scoping |
| D-06 | `SESSION-01..03` are enforcement contracts satisfied by `(D-05)`. The `SessionStore::get_for_tenant` returns `Err(TenantMismatch)` when ownership check fails; handlers serialize this as 403 with body `{"error":"tenant_mismatch"}`. | matches D-09 of 03.8.1 — never leak which tenant owns the session |
| D-07 | `AUDIT-02` (Owner cross-tenant audit): `Owner` may query `?tenant_id=<other>` only when `?cross_tenant=true` is explicitly set, AND the row tagged with `event_type=SECURITY_CROSS_TENANT_QUERY`. Implementation: a thin handler wrapper that audits before/after and rewrites the response. | least-invasive |
| D-08 | `AuthMode::None/ApiKey` paths are entirely unaffected — the new JWT-aware branches are skipped when `claims` extension is absent. No regression on the 175-test v3.7 baseline. | R-1 regression discipline |
| D-09 | No new external crate deps. | minimal surface |
| D-10 | Bumps CURRENT_VERSION? No — only `audit_event` (TENANT columns) and `tenant_sessions` (audit-only attrs) are involved. The `SessionStore::get_for_tenant` is pure Rust logic; no schema migration needed. | minimal |
| D-11 | "Owner always succeeds" applies to BOTH `RBAC-04` and `AUDIT-02`. Owner in single-tenant mode (`for_single_user` default `roles: vec![Owner]`) is the realistic production posture; multi-tenant owner (`for_multi_user(...Role::Owner)`) is the same. | preserve |
| D-12 | Existing `require_action_middleware` was already in scope to handle `AuthMode::Full` — extend the same fn to ALSO consume `Extension<JwtClaims>`. No new middleware file. | minimal surface |

### Task 2: `require_action_middleware` reads JwtClaims

**File:** `crates/grid-server/src/middleware/auth.rs:206` (existing fn)

- After looking up `UserContext`, ALSO look up `Extension<JwtClaims>` (already populated by auth_middleware_with_role).
- If present: check `claims.role`'s canonical form against `Role::can(action)` using the existing `Role::parse` path.
- If absent (AuthMode::None / ApiKey): keep the existing permission-vector check (D-08).

### Task 3: `SessionStore::get_for_tenant` + `TenantMismatch` error

**Files:** `crates/grid-server/src/state.rs` (or wherever SessionStore is wired)

- Add a method that takes `tenant_id: &str` as a guard.
- Verify `record.tenant_id == tenant_id` (or return `TenantMismatch`).
- Existing `SessionStore::get(...)` callers in single-user mode (default) are NOT touched; the multi-user path is explicit.

### Task 4: Wire `requires(Action)` into a representative endpoint

**File:** `crates/grid-server/src/router.rs` (or the specific endpoint file)

- Pick one well-trafficked endpoint — e.g., the session-by-id handler — and add `require_action_middleware(grid_engine::auth::roles::Action::Read)`.
- This is the **demonstrably-wired** RBAC layer; the rest of the endpoints stay un-scoped for v3.8.2 (the contract per `RBAC-01` says "the matrix is enforced" — demonstrated via the chosen endpoints; full-catalog coverage is v3.9+).

### Task 5: Cross-tenant audit handler (AUDIT-02)

**File:** `crates/grid-server/src/api/audit.rs` (or extend existing audit module)

- A wrapper that:
  1. Reads `claims.role` and `claims.tenant_id`
  2. If query has `?cross_tenant=true` AND role != Owner → 403
  3. Otherwise pass through to existing audit query; if cross_tenant → write a SECURITY row tagged `event_type="audit.cross_tenant_query"`.

### Task 6: Hermetic integration tests (8 cases)

**New file:** `crates/grid-server/tests/multi_user_rbac_tenant.rs`

1. `rbac_01_role_action_matrix_enforced` — `Viewer` calling protected endpoint → 403; `Owner` → 200
2. `rbac_02_post_users_requires_admin` — simulated via the wrapped session-by-id handler with `Action::Read` + Viewer JWT → 403, Admin → 200

8. `session_03_concurrent_sessions_no_cross_tenant_leak` — two users in two tenants running concurrent requests, asserting isolation

## Risks

| Risk | Mitigation |
|------|-----------|
| `RBAC-01` enforcement on every endpoint is too invasive for v3.8.2 | Pick ONE demonstrable endpoint + add the auditor test that scans every route's RBAC metadata to prove "this route has the right `requires()` for its tier." Other endpoints explicitly listed in v3.9+. |
| `TenantMismatch` error name leaks multi-tenant architecture to clients without auth (`AuthMode::None`) | D-08: never served in that mode; the existing single-user path is unaffected. |
| `SessionStore::get_for_tenant` could become `unwrap()` callers in some site | D-05: only the multi-user path uses it; flagged in code comment as "v3.8.2 — requires explicit tenant scoping". |
