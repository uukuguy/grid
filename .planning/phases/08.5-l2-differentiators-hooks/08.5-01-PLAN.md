    - Test 4: keyword-only mode (no query_embedding) — no crash, semantic_score stays 0.0
  </behavior>
  <action>
    Backfill `semantic_score` for FTS-only search hits using `embedding_vec` from the DB.

    **Current state**: In `index.py` `search()` method (lines 402-408), only memory IDs found in BOTH FTS and HNSW get `semantic_score` populated from HNSW cosine similarity. Memory IDs that appear in FTS but NOT in HNSW (because HNSW add previously failed) get `semantic_score=0.0`, producing misleading 0-score artifacts.

    **Implementation (per D95, ~40 LOC)**:
    1. After the FTS-only vs HNSW union merge (around line 430 where `results` is built), identify FTS-only hits: memory IDs in `fts_candidates` but NOT in `hnsw_by_id`.
    2. For each FTS-only hit, check if the query has a `query_embedding`. If so:
       a. Read `embedding_vec` blob from the DB row (it's available in the fetched row if `SELECT *` was used; otherwise do a targeted read).
       b. Use `unpack_embedding()` from `db.py` to convert the blob to `list[float]`.
       c. Compute cosine similarity between `query_embedding` and the unpacked vector.
       d. Clamp to [0, 1] (same as existing HNSW path at line 353).
       e. Set `semantic_score` on the `SearchHit`.
    3. If `embedding_vec` is NULL (legacy rows before embedding migration), keep `semantic_score=0.0` — no crash.
    4. If `query_embedding` is None (keyword-only mode), skip backfill — no crash.

    **Do NOT**: change the existing HNSW path (lines 402-408). HNSW scores take precedence when available. The backfill is only for gaps.
  </action>
