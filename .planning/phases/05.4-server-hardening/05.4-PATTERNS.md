```rust
pub fn migrate(conn: &Connection) -> rusqlite::Result<()> {
    conn.execute_batch("BEGIN EXCLUSIVE")?;  // file-level writer lock; serializes multi-process
    let version: u32 = conn.pragma_query_value(None, "user_version", |row| row.get(0))?;
    if version >= CURRENT_VERSION {
        conn.execute_batch("COMMIT")?;  // double-check exit: prior winner already migrated
        return Ok(());
    }
    info!(from = version, to = CURRENT_VERSION, "Running database migration (exclusive)");
    let migrations = get_migrations();

            info!("Applied migration v{}", migration.version);
        }
    }
    conn.pragma_update(None, "user_version", CURRENT_VERSION)?;
```

```rust
pub enum ConfigError {
    #[error("required env var {0} is not set")]
    MissingEnv(&'static str),
    #[error("unknown LLM_PROVIDER value: {0}")]
    UnknownProvider(String),
}

impl ProviderConfig {
    pub fn try_from_env() -> Result<Self, ConfigError> {
        let name = std::env::var("LLM_PROVIDER")
            .map_err(|_| ConfigError::MissingEnv("LLM_PROVIDER"))?;
        let (api_key_var, model_var) = match name.as_str() {
            "openai" => ("OPENAI_API_KEY", "OPENAI_MODEL_NAME"),
            "deepseek" => ("DEEPSEEK_API_KEY", "DEEPSEEK_MODEL_NAME"),
            "anthropic" => ("ANTHROPIC_API_KEY", "ANTHROPIC_MODEL_NAME"),
            other => return Err(ConfigError::UnknownProvider(other.into())),
        };
        let api_key = std::env::var(api_key_var)
            .map_err(|_| ConfigError::MissingEnv(api_key_var))?;
        let model = std::env::var(model_var)
            .map_err(|_| ConfigError::MissingEnv(model_var))?;
        let base_url = std::env::var(format!("{}_BASE_URL", name.to_uppercase())).ok();
        Ok(Self { name, api_key: Some(api_key), base_url, model: Some(model) })
    }
}
```

## Layering Conventions (cross-cutting)

These conventions apply across all NEW/MODIFIED files. Each `<read_first>` block in a plan task should reference whichever sub-section is relevant.

### LC-1 — `grid-engine::*` module layout

Modules under `crates/grid-engine/src/` are flat by capability (NOT by domain or layer). Existing examples: `audit`, `auth`, `mcp`, `metering`, `providers`, `memory`. Each module:

- Declares `pub mod` and re-exports top-level types in `lib.rs` (`crates/grid-engine/src/lib.rs:2-27` enumeration; `:37+` re-exports).
- Owns its own `mod.rs` + sub-modules (e.g. `audit/storage.rs`, `mcp/client.rs`).
- Is consumable from sibling crates as `grid_engine::<module>::<Type>`.

**5.4 new module `grid-engine::l2` must:**
- Live at `crates/grid-engine/src/l2/{mod.rs, client.rs}`.
- Be enumerated in `lib.rs` alphabetically (between `pub mod hooks;` and `pub mod logging;`).
- Re-export its public types at lib.rs top level (`pub use l2::{L2MemoryClient, ...}` mirroring `pub use audit::{AuditEvent, ...}` at `lib.rs:37`).

### LC-2 — `AppState::*_storage()` on-demand pattern

In `crates/grid-server/src/state.rs:87-101`, the convention is:

- AppState holds path/config data only (lightweight, `Send + Sync`).
- Each storage backend gets an `*_storage(&self)` method that builds a fresh client/connection per call. NEVER hold long-lived connections in AppState.
- Sync method when no DB open required (e.g. `mcp_storage`, `audit_storage` — sync because rusqlite::Connection::open is sync).
- `async fn` only when the backend needs `tokio-rusqlite` or async I/O (e.g. `metering_storage`).

**5.4 new `l2_storage()` must:**
- Be a sync method returning `grid_engine::l2::L2MemoryClient` (no `Option` wrapper — `from_env`/`new` never fail, just produces a client; failures occur at request time).
- Reuse `self.l2_client_base_url` field if present, else `from_env`.

### LC-3 — `RuntimeConfigOverrides` Some(field) → update pattern

In `crates/grid-server/src/api/config.rs:109-162`, each updatable field follows:


### LC-4 — Axum middleware LIFO layer order

`crates/grid-server/src/router.rs:264-289` shows the canonical chain:

- LIFO: last added = first to run.
- Desired order: `request_id` → `rate_limit` → `auth` → `audit` → handler.
- Audit MUST run after auth (so `UserContext` is in extensions).
- Each middleware uses `axum::middleware::from_fn_with_state(state.clone(), middleware_fn)`.

**5.4 changes do NOT alter the chain.** HMAC validation is added inside the existing `auth_middleware_with_role` function (§B4), not as a new layer.

### LC-5 — `tracing::info!` / `warn!` logging conventions

- DB migrations: `info!(from = ..., to = ..., "Running database migration")` per `db/mod.rs:33`.
- L2 errors: `warn!(error = %e, ...)` per `memory_write_hook.rs:90`. NEVER `error!` for L2 failures — they are non-fatal by design (`FailOpen`).
- WS lifecycle: `info!(session_id = %sid_str, ...)` per `ws.rs:258`.
