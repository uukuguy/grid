# Phase 3.7.1 — grid-cli 实战可用性补全 — Context

## Phase Boundary

`grid-cli` must let one person (non-developer observer) drive an agent end-to-end for **5 realistic enterprise scenarios** without manual CLI-flag tuning. Two-plan structure: Plan 01 audits the gap; Plan 02 closes the gap. Acceptance is a documented walkthrough record (new dated `docs/status/PRODUCTION_USABILITY_<date>.md`).

## Implementation Decisions

### 实战场景 — 验收基准 (locked 2026-07-19)

**D-01:** All 5 candidate enterprise scenarios are IN scope (per user selection). They form the acceptance baseline:

| # | Scenario | Acceptance marker |
|---|----------|-------------------|
| S1 | **Multi-step tool use** — agent reads a file, writes a file, calls a web API in sequence | PASS = each tool call observable in CLI output without `--debug` |
| S2 | **Memory-driven session** — agent recalls context from a prior turn (across session restart) | PASS = recall hit visible in output; `grid memory list` shows the written anchor |
| S3 | **Hook-driven governance** — PreToolUse hook fires; agent pauses for approval | PASS = hook decision prompt visible; approve path works without manual config tweaking |
| S4 | **Long-running streaming session with stop/resume** | PASS = `grid session run` streams chunks; Ctrl-C + `grid session resume <id>` recovers state |
| S5 | **Parallel agent runs / batch mode** | PASS = `grid run --parallel` (or equivalent) launches N agents; results aggregated |

**D-02:** Each scenario gets a **scenario walkthrough doc** under `docs/cli/scenarios/` (e.g. `S1-multi-step-tool-use.md`) covering: prerequisites, exact CLI invocation, expected output sketch, known gaps at audit time. These are inputs to Plan 01 and ground-truth for Plan 02 UAT.

### Quickstart 0→first-success 路径 (locked 2026-07-19)

**D-03:** Ship **both** paths (per user selection):
- **CLI path**: `grid quickstart <scenario>` subcommand that runs pre-flight (`grid init` equivalent checks), then executes S1 (the simplest scenario) end-to-end. Validates env vars, config, model reachability before scenario run.
- **Doc path**: `docs/cli/QUICKSTART.md` standalone doc with the same content for CTO/lead review.

**D-04:** `grid quickstart` is the **new CLI command** introduced in 3.7.1. Implementation lives in `crates/grid-cli/src/commands/quickstart.rs`. Default scenario = S1 (multi-step tool use). Other scenarios reachable via `grid quickstart S2` / `S3` / etc.

### Error recovery UX (locked 2026-07-19)

**D-05:** Adopt **fail-fast + `grid doctor --repair` suggestion** pattern (per user selection). Error output format:

```
error: <human-readable cause>
fix:   grid doctor --repair
```

**D-06:** Two error classes:
- **Retryable** (network/quota/transient) — fail-fast BUT error message says `transient, retry with --retry` flag (default off; user opts in).
- **Permanent** (config / permission / hook reject) — fail-fast, suggest `grid doctor --repair` or specific fix command.

**D-07:** TTY-aware fallback: when stderr is not a TTY (CI / scripted), output **machine-parseable JSON** (same fields as text, wrapped). When stderr is TTY, output human-readable text per D-05.

### Plan structure (locked 2026-07-19)

**D-08:** **Two plans, separate execution**:

- **Plan 3.7.1-01 (audit)**: For each of S1-S5, walk the scenario on a clean checkout with documented env vars. Output a gap audit table (`docs/audit/3.7.1-GAP-AUDIT.md`) with PASS / MANUAL TUNING / FAIL × scenario × command matrix. No code changes — read-only investigation. ~1-2 days.
- **Plan 3.7.1-02 (fix)**: Read audit output. For each FAIL/MANUAL TUNING row, derive a REQ-ID. Implement fixes, write scenario integration tests, ship `grid quickstart`. Output: 5 scenarios PASS + dated walkthrough record + UAT pass. Scope is bounded by audit findings; if findings exceed 1 plan, defer overflow to a Plan 03.

### the agent's Discretion

- **Audit scope depth** — Plan 01 decides which CLI subcommands to spot-check vs exhaustively run. (Reasonable default: spot-check S1-S5 + audit the 17-command surface in `commands/` for any obvious stubs.)
- **Walkthrough doc format** — `docs/cli/scenarios/S*.md` template is the agent's choice.
- **Doctor --repair coverage** — what fixes `--repair` knows about. (Existing A.3 audit reports 10 doctor checks; agent may add new repairable items.)
- **JSON error schema** — exact field names when in machine-parseable mode.

## Acceptance Standard

1. All 5 scenarios (S1-S5) PASS end-to-end on a clean checkout with documented env vars.
2. `grid quickstart <scenario>` runs S1 by default and the 5 named scenarios on demand.
3. `docs/cli/QUICKSTART.md` exists with mirrored content for non-CLI readers.
4. `docs/cli/scenarios/S{1..5}-*.md` walkthrough docs exist.
5. `docs/audit/3.7.1-GAP-AUDIT.md` exists with the audit matrix.
6. `docs/status/PRODUCTION_USABILITY_2026-XX-XX.md` dated walkthrough record exists with all 5 scenarios green.
7. No regression: `cargo test -p grid-cli` PASS, Activation-9.0 score preserved.

### Reusable Assets

### Established Patterns
- 17 subcommands organized in `commands/` (mod.rs + per-command .rs)
- Error type in `error.rs` — extend rather than replace
- Integration tests live in `tests/` (cli_integration.rs pattern)

### Integration Points
- `grid-cli` is `B only` (leg B per ADR-V2-024) — does not directly call EAASP tools; talks to `grid-server` via HTTP/WS
- S2 memory-driven session likely touches L2 memory via grid-server; S3 hook-driven touches hook infrastructure
- TUI (studio) and dashboard are SEPARATE entrypoints — 3.7.1 focuses on pure CLI path

## Specific Ideas

- S3 hook-driven governance scenario is the highest-value 实战 test because it exercises L3 governance hooks (per ADR-V2-019 deployment modes) which is what makes grid-cli "企业级" rather than toy.
- S4 stop/resume is the highest-risk scenario — `grid session resume <id>` recovery is a classic activation gap; this scenario surfaces it.
- The dated walkthrough record (`docs/status/PRODUCTION_USABILITY_<date>.md`) parallels the existing `docs/status/PRODUCT_STATUS_<date>.md` pattern — same audit-style format, frozen at sync moment.

- **`grid studio` / TUI as a quickstart entry point** — out of scope; TUI is its own surface, will be addressed if 3.7.1 reveals TUI gaps.
- **`grid` shell completions test scenario** — completions command exists; could be a S6, but not in current 5.
- **`grid eval` integration as S6** — A.3 audit already notes eval bridge was stubbed; if Plan 01 finds eval still stubby, that's an audit finding, not a new scenario.
- **`grid mcp` log streaming as S6** — A.3 audit noted MCP live log streaming was stubbed; same path as eval.
- **Bundle `grid quickstart` with `--tui` flag to launch into Studio TUI after scenario setup** — defer; out of scope for non-developer 0→first-success path.
