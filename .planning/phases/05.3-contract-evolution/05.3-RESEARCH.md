# Phase 5.3: Contract Evolution - Research

**Researched:** 2026-05-19
**Domain:** L1 runtime contract evolution (proto enum + Hook event additive bump) + ADR retroactive supersede + OpenAI-compat quirks framework
**Confidence:** HIGH

## Summary

Phase 5.3 ships contract-v1.2.0 across the L1 runtime stack ‚ÄĒ two additive
`ChunkType` wire values (`THINKING_TRACE`=8, `ATTACHMENT_REF`=9) and two
additive `HookEvent` variants (`SubagentStart`, `TaskCheckpoint`) ‚ÄĒ and closes
two contract-shaped deferred items (D109 invariant doc, D136 probe-turn hook
fire). Two new ADRs land in the same phase: ADR-V2-026 retroactively legitimizes
the `ExecutionMode` design already committed at `f1999fb`, and ADR-V2-027
formalizes an OpenAI-compat **quirks** abstraction (with prototype) so cross-
provider non-standard behaviors live in a single discoverable layer instead of
hot-fix soup in `openai.rs`.

All four contract surfaces are well-precedented: `CHUNK_TYPE_WORKFLOW_CONTINUATION=7`
was added incrementally in Phase 3.5 (commit `5cc0e4a`), `PreCompactHook` was
added incrementally in Phase 2 per ADR-V2-018, and the ADR governance plugin's
F1-F4 lint gate is already vendored under `.adr-plugin/scripts/` and exercised
by Phase 5.1 (ADR-V2-025). The whole-phase impact is mechanical and predictable;
the only design-shaped work is (1) THINKING_TRACE/TaskCheckpoint payload schemas
and (2) the quirks abstraction shape.

**Primary recommendation:** Split into 2 parallel-eligible plans ‚ÄĒ
**Plan A** (CONTRACT-01/02 + WATCH-01/03 + ADR-V2-021/006 increment) and
**Plan B** (ADR-V2-026 ExecutionMode + ADR-V2-027 Quirks + optional LingProvider).
B has zero hard code dependency on A ‚ÄĒ they touch disjoint files
(A: proto/, harness.rs hook sites, 7 runtime impl, mappers ; B: loop_config.rs,
openai.rs quirks refactor, deepseek.rs precedent, ADR docs). Parallelism saves
~30% wall time. See ¬ß"User Constraints" ‚Üí D-11 (no plan-split lock from user).

## User Constraints (from CONTEXT.md)

### Locked Decisions

**CONTRACT-01: ChunkType wire additions**
- **D-01:** Add BOTH `CHUNK_TYPE_THINKING_TRACE` (wire value 8) AND
  `CHUNK_TYPE_ATTACHMENT_REF` (wire value 9) in single contract-v1.2.0 release.
- **D-02:** `THINKING_TRACE` semantic = end-of-turn structured reasoning trace
  emitted right before MessageStop. Content is JSON:
  `{effort_level, total_reasoning_tokens, reasoning_summary?}`.
  Existing `CHUNK_TYPE_THINKING` (wire=2) stays = incremental SSE delta. Both
  can coexist in one turn. Downstream runtime is FREE to emit only one, both,
  or neither ‚ÄĒ capability, not requirement.
- **D-03:** `ATTACHMENT_REF` semantic = post-hoc pointer for blobs to avoid
  base64 bloat in SSE. Wire content is opaque envelope:
  `{"uri": "...", "mime_type": "...", "size": ...}`. **Backend (URI scheme
  whitelist + blob storage adapters) is OUT OF SCOPE for 5.3** ‚ÄĒ wire only
  carries the opaque URI string. Adding wire value now reserves the semantic.

**CONTRACT-02: Hook event additions**
- **D-04:** Add BOTH `SubagentStart` AND `TaskCheckpoint` events to
  `HookEvent.oneof event` in single contract-v1.2.0 release.
- **D-05:** `SubagentStart` semantic ‚ÄĒ fired when grid-engine
  `subagent_runtime.rs` spawns a sub-agent. Payload covers `parent_session_id`,
  `subagent_id`, `subagent_name`, `purpose?`. Phase 4.1 ¬ßF.Q3 audit item.
  L3 governance CAN deny via this hook (sub-agent security gate).
- **D-06:** `TaskCheckpoint` semantic ‚ÄĒ task boundary + checkpoint trigger
  needs sharper definition in RESEARCH.md (gray area: task boundary = user
  prompt? subtask in plan-tool? long-running tool exec?). 5.3 wire goes in,
  plan-phase researcher resolves the exact trigger criteria + content schema.

**WATCH-01 (D109): workflow.required_tools invariant doc**
- **D-07:** Document the invariant in
  `docs/design/EAASP/L1_RUNTIME_ADAPTATION_GUIDE.md` (per ROADMAP SC#3).
  Not a new ADR ‚ÄĒ invariant is sample-doc / spec-level, not architectural.

**WATCH-03 (D136): probe turn hook fire fix**
- **D-08:** Probe turn SHALL fire pre/post tool call hooks. The 3 contract
  xfails currently asserting "no fire in probe turn" flip to PASS. ADR-V2-016
  capability matrix section gets incremental update noting "probe turn IS
  observable to L3 governance + L2 trajectory, with optional
  `metadata.is_probe=true` flag for downstream filtering". Probe is a real
  LLM/tool call ‚ÄĒ invisibility is a bug, not a feature.

**ADR-V2-026: ExecutionMode supersede V2-016**
- **D-09:** RFC at `.planning/research/2026-05-16-agent-loop-execution-mode-rfc.md`
  + implementation at commit `f1999fb` is **locked as the design**. ADR-V2-026
  formalizes it retroactively. Default `AgentLoopConfig.execution_mode =
  Conversational`; `grid-runtime/main.rs` explicitly sets `LongWorkflow` for
  EAASP skill execution. No new code in this ADR ‚ÄĒ only the formal write-up
  + supersede chain to V2-016.
- **D-10:** No third "Auto" mode ‚ÄĒ explicit two-mode design, per anti-silent-
  fallback principle. Caller MUST declare intent.

**ADR-V2-027: OpenAI-compat quirks framework + prototype**
- **D-11:** Produce both **ADR + prototype impl** in 5.3. Avoids "principle
  ships, implementation doesn't follow" failure mode.
- **D-12:** F1 judgment rule: ‚Č•2 confirmed providers exhibit the same quirk
  ‚Üí it goes into `OpenAIProvider` quirks layer. Initial 3 quirks:
  - `reasoning_content_field` ‚ÄĒ deepseek/qwen/minimax/siliconflow
    (already hardcoded as multi-field scan at `openai.rs:834`)
  - `no_done_marker` ‚ÄĒ Ling-2.6-1T doesn't emit `data: [DONE]` (hot-fixed
    unconditionally at `Poll::Ready(None)` ‚ÄĒ needs quirk-gating to preserve
    assertion strength for compliant providers)
  - `null_tool_id_continuation` ‚ÄĒ ant-ling `tool_calls[].id=null` continuation
    pattern ‚ÄĒ promote IFF a second provider confirms; else fall to F2
    `LingProvider`
- **D-13:** F2 split rule: single-provider quirk AND behavior differs in ‚Č•3
  places ‚Üí split into dedicated provider (precedent: `deepseek.rs`). New
  `LingProvider` candidate IF `null_tool_id_continuation` stays single-provider.

### Pattern 1: ChunkType Wire Add (proven, Phase 3.5 precedent)

**What:** Add a new wire value to `ChunkType` enum across the entire stack
in a single PR.

**When to use:** Any new chunk semantic in `SendResponse.chunk_type`.

**Steps (in order):**

1. Edit `proto/eaasp/runtime/v2/common.proto` ‚ÄĒ append new `CHUNK_TYPE_X = N`
   line at the end of the enum block. **Do not renumber existing wires.**
2. Regen all 7 runtime stubs: `make build-eaasp-all` runs the proto codegen
   chain (Rust via `tonic-build` automatic; Python via `scripts/build_proto.py`;
   TS via hand-mirror ‚ÄĒ `lang/ccb-runtime-ts/src/proto/types.ts`).
3. Update `lang/ccb-runtime-ts/src/proto/types.ts` `export enum ChunkType` ‚ÄĒ
   add the two new variants with matching wire ints.
4. Run `scripts/check-ccb-types-ts-sync.sh` ‚ÄĒ must exit 0.
5. Emit at runtime sites: `chunk_type: ChunkType::CHUNK_TYPE_X as i32` per
   `crates/grid-runtime/src/service.rs` style (commit `e9472e4` is the
   reference: `chunk_type_to_proto()` mapper at service.rs boundary).
6. Verify L4 mapper picks it up: `_chunk_type_to_wire` is descriptor-driven
   so **no edit needed** ‚ÄĒ but a unit test must be added:
   `tools/eaasp-l4-orchestration/tests/test_chunk_coalescing.py::
   test_chunk_type_to_wire_known_variants` should be extended (precedent:
   line 439, asserts each enum int maps to the lowercase snake string).
7. Update CLI consumer: `tools/eaasp-cli-v2/src/eaasp_cli_v2/cmd_session.py`
   `_ALLOWED_CHUNK_TYPES` frozenset + corresponding `_render_chunk` branches.
8. Update guard test: `tests/contract/cases/test_chunk_type_contract.py`
   `ALLOWED_WIRE` frozenset (live-runtime parametrized case picks up new

## TaskCheckpoint Semantics (resolution of D-06 gray area)

Phase 5.3 CONTEXT.md D-06 explicitly defers the precise definition of
"task boundary" to RESEARCH.md. The recommendation here is intentionally
narrow to avoid over-design ‚ÄĒ we ship the wire and a minimal triggering
contract, deferring full task-graph reasoning to a later phase.

### Recommended scope for 5.3

**Trigger criteria (in priority order):**

1. **Explicit caller invocation** (HIGH confidence path) ‚ÄĒ L4 orchestration
   or a tool can explicitly request a checkpoint via a future
   `Checkpoint(session_id)` RPC. **Not in scope for 5.3** ‚ÄĒ but the wire is
   reserved so the hook event can be plumbed without proto-level rework.
2. **End of a successful tool call sequence where required_tools is fully
   satisfied** (RECOMMENDED for 5.3 initial impl) ‚ÄĒ fires when D87 Fix 2's
   "required_tools all satisfied" branch is taken at `harness.rs:1486`.
   This is a natural "skill workflow complete" boundary.
3. **MAX_WORKFLOW_CONTINUATIONS reached** (RECOMMENDED for 5.3) ‚ÄĒ fires
   when the LoopGuard fallback triggers; gives L2/L3 a chance to capture
   the partially-complete state.

**Explicitly NOT triggers for 5.3:**

- Every user prompt (too noisy; "session_start" already covers session-
  level boundary)
- Every tool call (PreToolUse/PostToolUse already cover this)
- Token-budget checkpoint (PreCompactHook ADR-V2-018 already covers this)

### Payload schema

```proto
message TaskCheckpointHook {
  string reason = 1;                  // "required_tools_satisfied" | "max_continuations_reached" | "explicit"
  uint32 rounds_completed = 2;        // agent loop rounds at checkpoint
  uint32 total_tool_calls = 3;        // total tool calls executed in session
  repeated string completed_tools = 4; // tool names that have been satisfied
  string snapshot_uri = 5;            // OPTIONAL ‚ÄĒ reserved for ATTACHMENT_REF integration (future blob storage)
}
```

**Confidence:** [ASSUMED] for trigger criteria ‚ÄĒ these are best-guess from
the existing agent loop control flow; D-06 explicitly defers exact semantics
to RESEARCH and planner can pick another shape. Worth confirming with user

field detection; [ASSUMED] for the specific `effort_level/total_reasoning_tokens/reasoning_summary`

## Common Pitfalls

### Pitfall 1: Forgetting TS-side ChunkType sync

**What goes wrong:** Add wires to common.proto, regen Rust + Python stubs,
forget to update `lang/ccb-runtime-ts/src/proto/types.ts`. CI exit-1 on
`scripts/check-ccb-types-ts-sync.sh`.

**Why it happens:** ccb-runtime-ts uses dynamic JSON message loading via
`@grpc/proto-loader`, but the `ChunkType` enum is hand-mirrored. D149
exists specifically because this drift happened once.

**How to avoid:** Plan-phase task block MUST include "edit types.ts" as
explicit step. CI gate catches it pre-merge.

**Warning signs:** PR diff doesn't touch `lang/ccb-runtime-ts/src/proto/types.ts`.

### Pitfall 2: Renumbering existing ChunkType wire values

**What goes wrong:** Insert new wire as `CHUNK_TYPE_THINKING_TRACE = 2` and
shift existing values. Breaks every Rust/Python/TS binary compiled against
the old proto.

**Why it happens:** "It makes more sense semantically" ‚ÄĒ but proto3 closed
enums are ordered by wire int, not by semantic adjacency.

**How to avoid:** ALWAYS append at end. Current last value is
`CHUNK_TYPE_WORKFLOW_CONTINUATION = 7`. New values are 8 and 9.

**Warning signs:** PR touches any line of the proto enum other than appending
two new lines.

### Pitfall 3: Hook envelope shape regression on SubagentStart

**What goes wrong:** Define `SubagentStartHook` payload with nested
`payload.subagent_id` (nested struct). Hook scripts reading the envelope
fail because ADR-V2-006 ¬ß2.3 requires top-level keys.

**Why it happens:** Phase 5.0 D134 specifically closed this for existing
hooks; new hook authors may not know the ¬ß2.3 invariant.

**How to avoid:** Match the existing pattern in `PreToolUse` (top-level
`tool_name`, `tool_args`) rather than introducing nesting. SubagentStart
envelope should be:

```json
{
  "event": "SubagentStart",
  "session_id": "<parent>",
  "subagent_id": "<child>",
  "subagent_name": "<task name>",
  "purpose": "<short reason>",
  "skill_id": "",
  "created_at": "..."
}
```

**Warning signs:** Any `payload.*` keys appearing in the new HookContext
extension.

### Pitfall 4: ADR-V2-026 lint F4 cross-module conflicts

**What goes wrong:** ADR-V2-026 declares `affected_modules:
[crates/grid-engine/, crates/grid-runtime/, crates/grid-cli/, crates/grid-server/]`.
F4 lint fails because ADR-V2-016 also lists `crates/grid-engine/` ‚ÄĒ
multiple Accepted ADRs covering the same module = F4 conflict.

**Why it happens:** F4 lint enforces no-overlap to ensure single-ADR
ownership per module. Phase 5.1 V2-025 acceptance had to bypass F4 (noted
in SUMMARY).

**How to avoid:** Two paths:
1. Mark V2-016 `superseded_by: ADR-V2-026` BEFORE running F4 on V2-026 ‚ÄĒ
   the lint may skip superseded ADRs (verify with `.adr-plugin/scripts/adr-lint.sh`).
2. Bypass F4 with explicit `--allow-conflict` flag (Phase 5.1 / ADR-V2-021
   precedent) and document in PR description.

**Warning signs:** F4 lint output mentions "module overlap with ADR-V2-016".

### Pitfall 5: Quirks unconditionally promoted from hot-fix

**What goes wrong:** `no_done_marker` is currently unconditional at
`openai.rs:981-1018`. If we promote it to a quirk WITHOUT making it gated,
then EVERY OpenAI-compat provider gets the loose stream-end handling, which
masks legitimately broken streams (e.g., network truncation before [DONE]).

**Why it happens:** Hot-fix-driven development ‚ÄĒ the fix was applied
universally to unblock Ling-2.6, then "looks fine in testing" makes it
seem safe to keep universal.

**How to avoid:** ADR-V2-027 prototype MUST default `quirks.no_done_marker
= false`, and explicit caller (LingProvider or yaml config) must opt-in.
Default OpenAI/DeepSeek behavior keeps the strict assertion (no [DONE] =
suspicious stream).

**Warning signs:** Diff to `openai.rs:981-1018` removes the conditional
gate without adding an explicit `if quirks.no_done_marker { ... }` wrap.

### Pitfall 6: Conflating D136 "probe" with capability_probe

**What goes wrong:** Read "D136 probe-turn hook fire fix" and start
investigating `capabilities.rs::probe_tool_choice`. Waste hours.

**Why it happens:** "probe" is overloaded ‚ÄĒ D136 means the contract test
probe-skill fixture invocation; the capability matrix probe is a separate
concept that ALSO benefits from `is_probe=true` metadata but is not the
target of the D136 xfail flip.

**How to avoid:** Read DEFERRED_LEDGER ¬ßD136 fully before diving into code.
The actual fix lives in `tests/contract/harness/mock_openai_server.py` not
in any grid-engine module.

**Warning signs:** Plan-phase task description says "find probe_tool_choice
and add is_probe field" ‚ÄĒ that's not the D136 fix, that's the optional
ADR-V2-016 amendment companion work.
