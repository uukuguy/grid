## Decisions Made

See frontmatter `key-decisions` for the 5 substantive design decisions made during execution. The most consequential:

1. **SubagentStart fire location** — at the head of the spawned task rather than before `tokio::spawn`. This preserved `run_async`'s sync signature without pulling in the `futures` crate. The L3 governance check still completes before any LLM/tool work begins, so the observable semantics ("pre-spawn deny") match the plan's intent.

2. **TaskCheckpoint payload via metadata HashMap** — caller populates `HookContext.set_metadata("reason", Value::String(...))` etc., the `to_task_checkpoint_envelope` emitter does metadata→top-level JSON projection. Avoided adding 4 new typed fields to HookContext for a 2-trigger MVP; future trigger growth happens at the emitter, not the struct.

3. **D136 fix via Option A (mock parity with production parser)** — preserved production assertion strength rather than loosening the parser to match a weak fixture (RESEARCH §F2 Option B).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] SubagentStart fire site changed from "before tokio::spawn with futures::executor::block_on" to "at the head of the spawned task"**
- **Found during:** Task 5.3-01-03 (SubagentStart fire wiring)
- **Issue:** Plan's literal action body called for `futures::executor::block_on(self.hooks.execute(...))` before `tokio::spawn`. The `futures` crate is not in the grid-engine `Cargo.toml` (only `futures-util` is, and it does not expose `executor::block_on`). Adding the `futures` crate as a workspace dependency would be a scope-expanding change.
- **Fix:** Moved the hook fire to the head of the spawned task body. Because the hook call is the very first `.await` inside the spawned closure, no agent_loop / LLM / tool work happens before L3 governance sees the event. The Block/Abort path calls `mgr.fail` and returns from the task; no agent_loop is invoked.
- **Files modified:** `crates/grid-engine/src/agent/subagent_runtime.rs`
- **Verification:** `cargo test -p grid-engine --test subagent_start_hook_test --test-threads=1` → 2/2 PASS, including the `subagent_start_block_prevents_spawn` test that uses a `MarkerProvider` whose `stream()` would panic if called.
- **Committed in:** `c33f122` (Task 3 commit)

**2. [Rule 2 - Missing Critical] Added a new "max_continuations_reached" pre-D87 guard branch in harness.rs**
- **Found during:** Task 5.3-01-04 (TaskCheckpoint fire wiring)
- **Issue:** Plan asked for a fire site at "the MAX_WORKFLOW_CONTINUATIONS branch", but the existing harness.rs code had no explicit "count reached" branch — it just stopped continuing when `workflow_continuation_count >= MAX_WORKFLOW_CONTINUATIONS` (via the `< MAX` condition at line 1461). A silent fall-through wouldn't fire the hook.
- **Fix:** Added an explicit guard block immediately before the D87 Fix 2 conditional that fires TaskCheckpoint with `reason="max_continuations_reached"` when the harness would otherwise continue but the count has been exhausted. Then falls through to the normal exit branch with no behavior change to the continuation logic itself.
- **Files modified:** `crates/grid-engine/src/agent/harness.rs`
- **Verification:** `cargo test -p grid-engine --test task_checkpoint_hook_test --test-threads=1` → 2/2 PASS, including `task_checkpoint_fires_on_max_continuations_reached`.
- **Committed in:** `c045584` (Task 4 commit)
