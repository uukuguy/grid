<decisions>
## Implementation Decisions

### CONTRACT-01: ChunkType wire additions
- **D-01:** Add BOTH `CHUNK_TYPE_THINKING_TRACE` (wire value 8) AND
  `CHUNK_TYPE_ATTACHMENT_REF` (wire value 9) in single contract-v1.2.0 release.
- **D-02:** `THINKING_TRACE` semantic = **本轮一次性汇总的结构化 reasoning
  trace**, emitted at end of turn (typically right before MessageStop). Content
  is JSON: `{effort_level, total_reasoning_tokens, reasoning_summary?}`.
  Existing `CHUNK_TYPE_THINKING` (wire value 2) keeps current semantic =
  **incremental reasoning stream during SSE**. Both can coexist in one turn:
  THINKING deltas during streaming + one THINKING_TRACE at end. Downstream
  runtime is FREE to emit only one, both, or neither — this is a capability,
  not a requirement.
- **D-03:** `ATTACHMENT_REF` semantic = **post-hoc attachment pointer to avoid
  base64 bloat in SSE**. Wire content is opaque URI envelope:
  `{"uri": "...", "mime_type": "...", "size": ...}`. **Backend (URI scheme
  whitelist + blob storage adapters) is OUT OF SCOPE for 5.3** — wire just
  carries the opaque URI string, L4 / L2 / downstream interpret it (likely
  candidates: `file://`, `s3://`, `l2-memory://`). Adding wire value now reserves
  the semantic; backend lands next milestone.

### CONTRACT-02: Hook event additions
- **D-04:** Add BOTH `SubagentStart` AND `TaskCheckpoint` Hook events to
  `proto/eaasp/runtime/v2/hook.proto` `HookEvent.oneof event` in single
  contract-v1.2.0 release.
- **D-05:** `SubagentStart` semantic — fired when grid-engine
  `subagent_runtime.rs` spawns a sub-agent. Payload covers `parent_session_id`,
  `subagent_id`, `subagent_name`, `purpose?`. Phase 4.1 §F.Q3 audit item.
  L3 governance CAN deny via this hook (sub-agent security gate).
- **D-06:** `TaskCheckpoint` semantic — task boundary and checkpoint trigger
  needs sharper definition in RESEARCH.md (gray area: task boundary = user
  prompt? subtask in plan-tool? long-running tool exec?). 5.3 wire goes in,
  plan-phase researcher resolves the exact trigger criteria + content schema.

### WATCH-01 (D109): workflow.required_tools invariant doc
- **D-07:** Document the invariant in `docs/design/EAASP/L1_RUNTIME_ADAPTATION_GUIDE.md`
  (per ROADMAP SC#3). Not a new ADR — invariant is sample-doc / spec-level,
  not architectural decision. ADR-V2-020 (skill-declared tool filter) is a
  different layer (L2) so do not bundle.

### WATCH-03 (D136): probe turn hook fire fix
- **D-08:** Probe turn SHALL fire pre/post tool call hooks. The 3 contract
  xfails currently asserting "no fire in probe turn" flip to PASS. ADR-V2-016
  capability matrix section gets an incremental update noting "probe turn IS
  observable to L3 governance + L2 trajectory, with optional
  `metadata.is_probe=true` flag for downstream filtering". Rationale: probe is
  a real LLM/tool call — making it invisible to observability is a bug, not a
  feature. The silly scenario "L3 denies the probe" is handled by L3 being
  smart about `is_probe=true` metadata rather than by hiding the call.

### ADR-V2-026: ExecutionMode supersede V2-016 (NEW-E4 → Accepted)
- **D-09:** RFC at `.planning/research/2026-05-16-agent-loop-execution-mode-rfc.md`
  + implementation at commit `f1999fb` is **locked as the design**. ADR-V2-026
  formalizes it retroactively. Default `AgentLoopConfig.execution_mode =
  Conversational`; `grid-runtime/main.rs` explicitly sets `LongWorkflow` for
  EAASP skill execution. No new code in this ADR — only the formal write-up
  + supersede chain to V2-016.
- **D-10:** No third "Auto" mode — explicit two-mode design, per anti-silent-
  fallback principle. Caller MUST declare intent.

### ADR-V2-027: OpenAI-compat quirks framework + prototype (NEW-F1/F2)
- **D-11:** Produce both **ADR + prototype impl** in 5.3. Splitting into "ADR
  in 5.3 + impl in 5.4" risks "principle ships, implementation doesn't follow"
  — make decision-and-flesh same phase.
- **D-12:** F1 judgment rule: ≥2 confirmed providers exhibit the same quirk →
  it goes into `OpenAIProvider` quirks layer. Initial 3 quirks to standardize
  (all confirmed in this session 2026-05-19):
  - `reasoning_content_field` — deepseek/qwen/minimax/siliconflow share
    `reasoning_content` delta field (currently hardcoded in openai.rs:834,
    promote to quirks config)
  - `no_done_marker` — Ling-2.6-1T doesn't emit `data: [DONE]` (hot-fixed
    2026-05-19 unconditionally in `Poll::Ready(None)`; should become
    quirk-gated to preserve assertion strength for compliant providers)
  - One more confirmed in second provider TBD (parser hints: `tool_calls[].id`
    null continuation pattern from ant-ling — needs ONE more provider
    confirmation before promotion; if not, falls to F2)
- **D-13:** F2 split rule: a quirk only confirmed in 1 provider AND
  behavior-differs-from-OpenAI in ≥3 places (or breaks common-path
  complexity) → split into dedicated provider type (precedent:
  `crates/grid-engine/src/providers/deepseek.rs`). Phase 5.3 produces
  `LingProvider` as 第二 reference of this pattern, IF the `null_tool_id_
  continuation` quirk stays single-provider. The deepseek-reasoner
  `reasoning_content` history-required behavior is the next strong split
  candidate, but it lives in deepseek's existing provider so already split.

<specifics>
## Specific Ideas

- **Bundling rationale**: NEW-E4 (ExecutionMode ADR-V2-026) bundled into 5.3
  not because it's contract-shaped but because (a) RFC + impl already ready
  so cost is low, (b) clean ADR numbering (V2-026 + V2-027 in same phase),
  (c) keeps ADR-V2-016 supersede chain visible in same commit cluster as
  contract-v1.2.0 bump.
- **NEW-F1/F2 timing rationale**: User direction 2026-05-19 was "shared
  non-standard behaviors live in OpenAIProvider; provider-unique quirks split
  into their own provider crate". Doing ADR + prototype in same phase avoids
  the "ADR ships, code doesn't follow" failure mode user explicitly flagged
  ("你很爱用 fallback, 出问题找都没法找"). NEW-F3 (anti-silent-fallback)
  cluster goes to 5.4 since it touches LLM_PROVIDER / RUST_LOG / logger /
  OPENAI_NO_PROXY — same fail-fast theme as 5.4's auth+audit+config strictness.
- **D136 framing**: Probe turn = real LLM/tool call → MUST be observable.
  Hiding it from L3/L2 is a bug-not-feature. If L3 governance over-denies
  probes, fix L3 (add is_probe-aware policy), don't hide the event.
