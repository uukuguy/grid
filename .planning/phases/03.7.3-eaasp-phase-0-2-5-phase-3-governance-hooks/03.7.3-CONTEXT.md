# Phase 3.7.3 — EAASP 本地仿真补全 — Context (Decisions Locked)

## Phase Boundary

Move `tools/eaasp-*/` from "simulator-level reference implementation" (per `docs/PROJECT_PRODUCT_OVERVIEW.md`) to "credible enterprise simulation close enough to 实战 that a customer could evaluate the platform locally before committing to deployment". Specifically: wire **minimum credible Phase 3 governance gate hooks** — risk-classified actions must actually **pause for approval** rather than silently pass — so a realistic enterprise workflow (agent → writes to external system → governance gate → user approves → action completes) runs end-to-end through EAASP local tools with observable governance behavior.

**Full Phase 3 production OPA approval chain remains future work** (per `docs/design/EAASP/EAASP_v2_0_EVOLUTION_PATH.md` §3.1 — Phase 3 ⏸); 3.7.3 wires the in-process hook layer that makes Phase 3 *demonstrable*, not the production OPA backend that the hook layer will eventually call.

**Validation standard** (per ROADMAP.md §3.7.3 Validate): 1 实战 enterprise scenario (e.g. agent writes to external system, governance gate triggers, user approves, action completes) runs end-to-end through EAASP local tools with observable governance behavior.

## Implementation Decisions

### Risk classification & hook point

- **D-01 Risk taxonomy**: per-tool risk metadata declared in **skill manifest** (already SHIPPED in Phase 2 via `tools/eaasp-skill-registry`). Three levels: `read` (auto-allow), `write_local` (shadow), `write_external` (gate). Skill manifest gets a new optional `risk_level` field; default `read`. L3 policy can override per-hook via existing `ManagedHook.risk_level` field.
- **D-02 Hook enforcement point**: **L3 policy-engine** as gatekeeper, intercepting after tool invocation resolves but before the actual side effect executes. L4 orchestration consumes the resulting `governance.decision` event and forwards via existing SSE channel. L1 runtime stays protocol-pure (no L3 awareness); aligns with Phase 3 design intent (`docs/design/EAASP/PHASE_3_DESIGN.md`).
- **D-03 L3 audit extension**: extend `tools/eaasp-l3-governance/src/eaasp_l3_governance/audit.py` with a new `governance_decisions` table — captures `{decision_id, session_id, hook_id, tool_name, risk_level, decision (allow|approve|deny), approver, rationale, ts}`. Backward compatible: existing audit entries remain unchanged.

### Approval UX

- **D-04 CLI approval UX**: **synchronous prompt** for `eaasp session run` (CLI is the primary entrypoint per 3.7-CONTEXT.md §实战可用 acceptance standard #3). `--yes` flag auto-approves for non-interactive use (mirrors `apt -y`, `kubectl apply -f` patterns); `--no` flag auto-denies.
- **D-05 L4 SSE event**: when a gate triggers during l4 session execution, emit `governance.request` event (payload: `{decision_id, hook_id, tool_name, risk_level, action_preview}`) into the SSE stream. External consumers (web dashboard, IM bot — future work) can react. For 3.7.3 scope: SSE emission only; consumers are out of scope.
- **D-06 No new HTTP endpoint in L3**: approval gate is invoked **in-process** by `eaasp-cli session run` against L3 (via existing `client.call` channel). Rationale: keeps scope minimal; L3 already has policy-deploy + switch-mode endpoints; adding approval HTTP endpoint would expand 3.7.3 scope. Full Phase 3 OPA approval chain (HTTP approval endpoints, multi-party approval, audit retention) remains future work.

### Backward compat

- **D-07 Default mode preserved**: existing `ManagedHook.mode ∈ {shadow, enforce}` behavior **unchanged**. New behavior is a parallel **risk-level gate** that activates only when:
  1. Skill manifest declares `risk_level ∈ {write_local, write_external}`, AND
  2. L3 policy does NOT have a `shadow` mode override for the hook (i.e. real enforcement).
- **D-08 Existing tests unaffected**: all 12 existing L3 tests + all 14 EAASP Python runtime tests + all 5 cli-v2 tests must pass unchanged. New tests (Plan 02) cover the gate logic in isolation.

### Validation scenario

- **D-09 Demo scenario**: extend the `mock-scada` example (`tools/mock-scada/` already SHIPPED) with a scenario `S8 — Risk-classified external write`. Script: agent asks LLM to "update SCADA setpoint X to value Y" → skill tool `scada_set_setpoint` has `risk_level: write_external` → L3 gate triggers → CLI prompts `Approve? [y/N]` → user approves → tool executes → `governance.decision` event flows through SSE → audit row recorded. **Single scenario** as ROADMAP.md §3.7.3 spec requires.

## Specific Ideas

- **Naming convention**: `governance_decision` table (singular noun in lowercase, snake_case). New SSE event types: `governance.request` (gate triggered, awaiting user) and `governance.decision` (user approved/denied).
- **CLI prompt copy**: mirror `apt` / `yum` style — show what will happen, ask for confirmation. Example:
  ```
  [governance] Risk-classified action requested:
    tool:   scada_set_setpoint
    risk:   write_external
    effect: will write to 192.168.1.100:5020 (SCADA gateway)
  Approve? [y/N]:
  ```
- **Audit row example**:
