# Phase 7.3: L3 RBAC + hardening - Context

**Gathered:** 2026-06-07
**Status:** Ready for planning

<domain>
## Phase Boundary

L3 governance (`tools/eaasp-l3-governance/`) is the v3.3 milestone's last phase. It currently has **zero authentication, zero RBAC, no global exception handler, no logging, and unresolved P3 tech-debt**. This phase lands the 3 P2 items (D8/D9/D46) and cleans up the 5 P3 items (D17/D18/D22/D23/D26) to bring L3 to production-readiness.

Specifically:

- **D8 (P2)** — `access_scope` 真实 RBAC 中间件：session token carries scope, L3's FastAPI middleware/dependency rejects calls before they reach route handlers.
- **D46 (P2)** — Skill namespace 校验：skill-registry (`tools/eaasp-skill-registry/src/store.rs`) rejects registration when the declared `access_scope` conflicts with an already-deployed skill's namespace.
- **D9 (P2)** — `skill_usage` 真实遥测：NEW `GET /v1/telemetry/skill-usage` endpoint. Primary path: query L2's audit log via HTTP. Fallback: aggregate from L3's own `telemetry_events` table. Returns per-skill invocation counts.
- **D22 (P3)** — 全局 FastAPI exception handler：`@app.exception_handler(Exception)` wraps all unhandled exceptions into JSON `{"error": "internal_error", "detail": str(e)}` per ADR-V2-028 strict-by-default lineage.
- **D23 (P3)** — loguru 初始化：configure `loguru.logger` in `create_app()` lifespan with stderr sink + structured format. Add `loguru` to `pyproject.toml` dependencies.
- **D17 (P3)** — `hook["hook_id"]` KeyError guard：replace bare dict access with `.get("hook_id")` + raise typed `HTTPException(422, ...)` on missing key.
- **D18 (P3)** — `session_id` path param validation：add Pydantic `Field(min_length=1)` or UUID regex to `session_id` parameter; reject empty/unparseable session IDs with 422.
- **D26 (P3)** — 消除 `asyncio.sleep(1.1)`：add `_tiebreaker: int` field to `TelemetryEventIn`, auto-increment per-test. Tests inject explicit tiebreaker values for reproducible ordering. Clock-independent.

**Out of scope (intentional ceilings):**
- D10 (eval frontmatter baseline owner) — defer to v3.4+
- D16 (file_upload route impl) — defer to v3.4+
- D19/D20/D21 (split out eval_manager / separate serving / PolicyEngine cross-contamination) — defer to v3.4+
- D25 (pydantic v2 `globs` validator migration) — defer to v3.4+
- D31 (L4 loguru init — same D23 pattern but separate phase)
- Full JWT/OAuth2 auth infrastructure — this phase uses a lightweight header-based scope token. Proper identity provider integration is deferred to v3.4+.
- Production-grade L2 HTTP client with connection pooling — the D9 L2-audit query uses a simple `httpx.AsyncClient` single-request pattern. Pooling + retries deferred.
- Multi-tenant scope enforcement — single-scope-per-token model for v3.3.

<decisions>
## Implementation Decisions

### D-01 — RBAC strategy: FastAPI middleware + Depends (D8) + skill-registry namespace guard (D46)

**Decision:** Two enforcement points:

1. **L3 runtime RBAC (D8)**: FastAPI `Depends()` dependency extracts `X-Session-Scope` header from incoming requests. If the scope doesn't match the target resource's `access_scope`, the dependency raises `HTTPException(403, ...)`. The dependency is injected into `validate_session()` and any future scope-sensitive endpoints. This is NOT a global middleware — it's a reusable `Depends()` function that each scope-sensitive route explicitly includes.

2. **Skill-registry namespace guard (D46)**: Implemented in `tools/eaasp-skill-registry/src/store.rs`. During skill `submit()`, parse the v2 frontmatter's `access_scope`, query the `skills` table for ANY existing skill whose `access_scope` equals the new skill's scope. If a conflict exists AND the existing skill has a different `id`, the registry rejects the submission with a descriptive error (`namespace_conflict: {scope} already claimed by skill {existing_id}`). The guard allows re-registration of the same skill (same `id` + same scope) since that's an update/version bump.

**Auth token model** (lightweight for v3.3):
- Header: `X-Session-Scope: <scope_name>` (e.g., `ecommerce`, `finance`, `*` for wildcard)
- The dependency function `require_access_scope(required: str)` returns the scope or raises 403.
- `validate_session()` calls `require_access_scope(hook["access_scope"])` for each hook that declares a scope.
- No HMAC/signing at this stage — L3 is an internal service behind EAASP's gateway. Full token auth deferred to v3.4+ (see Deferred).

### D-02 — Telemetry strategy: L2-primary + L3-fallback (D9)

**Decision:** New `GET /v1/telemetry/skill-usage?skill_id=<id>&since=<ISO>` endpoint:

1. **Primary path**: Make HTTP GET to L2's memory-engine audit endpoint (`GET /v1/memory/{skill_id}/usage?since=...` — TBD exact path, scout during plan-phase). If L2 responds with 2xx, return the aggregated data directly.
2. **Fallback path**: If L2 is unreachable (connection error, timeout, 5xx), fall back to L3's own `telemetry_events` table: `SELECT COUNT(*) as invocations, MIN(received_at) as first_seen, MAX(received_at) as last_seen FROM telemetry_events WHERE skill_id = ? GROUP BY skill_id`.
3. **Response shape**: `{"skill_id": ..., "invocations": N, "first_seen": "...", "last_seen": "...", "source": "l2" | "l3_fallback"}`.
4. **L2 client**: `httpx.AsyncClient` (already a dependency of L3) with a configurable base URL via env `L2_MEMORY_ENGINE_URL` (default `http://localhost:18082`). Timeout: 5s for primary path.
5. **Test strategy**: Unit test with mocked httpx transport (L2-primary path) + integration test with real L3 DB (fallback path). L2-primary path test uses `pytest-httpx` or manual `httpx.MockTransport`.

**Why not query L2 exclusively**: L2 may not be deployed in all EAASP configurations (some deployments run L3 standalone). The fallback keeps `skill_usage` functional even without L2, with `source: "l3_fallback"` making the data source transparent.

### D-03 — Error handling + logging: JSON envelope + loguru (D22/D23)

**Decision:**

1. **D22 — Global exception handler**: Add `@app.exception_handler(Exception)` in `create_app()` that catches all unhandled exceptions and returns:
   ```json
   {"error": "internal_error", "detail": "<sanitized message>"}
   ```
   - Status code: 500 unless the exception has a `status_code` attribute (e.g., HTTPException subclasses).
   - Sanitization: `str(exc)` with 500-char max to prevent stack trace leakage.
   - Pydantic `ValidationError` gets special treatment: 422 with field-level error details (preserves existing `_sanitize_errors()` behavior).

2. **D23 — loguru initialization**: In `create_app()` lifespan startup:
   ```python
   from loguru import logger
   logger.remove()  # clear default
   logger.add(
       sys.stderr,
       format="{time:ISO} | {level: <8} | {name}:{function}:{line} | {message}",
       level=os.environ.get("L3_LOG_LEVEL", "INFO"),
   )
   ```
   - Add `loguru>=0.7` to `pyproject.toml`.
   - Logger instance stored as module-level `logger` (standard loguru convention).
   - Instrument key decision points: policy deploy, scope rejection, L2 query failure, session validation miss.

### D-04 — Test timing: monotonic tiebreaker (D26)

**Decision:**

1. Add `_tiebreaker: int = 0` field to `TelemetryEventIn` Pydantic model in `audit.py`. Default is 0 (production code never sets it; two events with same tiebreaker are ordered by `received_at` — acceptable for production).
2. `AuditStore.ingest()` stores `_tiebreaker` in the `telemetry_events` table alongside existing columns.
3. In tests, inject explicit tiebreaker values: `TelemetryEventIn(session_id=..., hook_data=..., _tiebreaker=1)` then `... _tiebreaker=2)`.
4. Replace `asyncio.sleep(1.1)` in `test_query_newest_first` and `test_query_since_filter` with tiebreaker-based ordering assertions.
5. Add `_tiebreaker INTEGER NOT NULL DEFAULT 0` column to schema migration (idempotent `ALTER TABLE` with `PRAGMA table_info` probe).
6. No test dependency changes (no `freezegun`, no `time_machine`).

### D-05 — D17 fix scope: `.get()` with typed error

**Decision:** Replace `hook["hook_id"]` at `api.py:136` with:
```python
hook_id = hook.get("hook_id")
if hook_id is None:
    raise HTTPException(
        status_code=422,
        detail={"error": "invalid_hook", "message": "hook missing required field: hook_id"}
    )
```
Single-line change, no new types, no Pydantic model for the hook dict (keeps it lightweight).

### D-06 — D18 fix scope: path param validation

**Decision:** Add validation to the `session_id` path parameter in `validate_session()`:
```python
@router.post("/v1/sessions/{session_id}/validate")
async def validate_session(
    session_id: str = Path(..., min_length=1, pattern=r"^[a-zA-Z0-9_-]+$"),
    body: SessionValidateRequest = Body(...),
) -> dict[str, Any]:
```
FastAPI's `Path()` with `min_length=1` rejects empty strings automatically (422). The `pattern` regex allows alphanumeric + hyphens + underscores — matches typical session ID formats. No UUID enforcement (session IDs come from EAASP's L4 orchestrator, which uses varied formats).

### D-07 — Plan structure: single plan (≤8 tasks)

**Decision:** One plan for Phase 7.3. All 8 items are tightly coupled to `api.py` / L3 infrastructure. Splitting would create artificial dependency ordering and force task sequencing that's simpler in a single plan. Task count: 8 REQ-IDs → 8 tasks (one per REQ-ID), which fits within the ≤10 task ceiling.

**Task ordering (DAG):**
```
Wave 0 (infra):  D23 (loguru) → D22 (exception handler, depends on logger)
Wave 1 (auth):   D46 (skill-registry guard, independent) ∥ D08 (RBAC middleware, after D22)
Wave 2 (telemetry): D09 (skill_usage, after D08 for RBAC on new endpoint)
Wave 3 (cleanup): D17 (hook_id guard) ∥ D18 (session_id validation) ∥ D26 (test tiebreaker)
```

### D-08 — Auth token: `X-Session-Scope` header, no signing

**Decision:** `X-Session-Scope: <scope>` header with a single scope value. The `require_access_scope()` dependency extracts this header, compares it to the required scope, and returns 403 on mismatch. For the `*` wildcard scope (admin/debug), all checks pass.

**Why not JWT/HMAC**: v3.3 is a focused debt sweep, not a new feature development milestone. Full auth infrastructure (identity provider, token signing, key rotation) would be a 2-3 phase effort. A lightweight header-based scope token is sufficient for L3's current role as an internal service behind EAASP's L4 gateway. Proper auth is filed as a deferred item for v3.4+.

</decisions>

- Scope header: missing on scope-guarded endpoints → 403, not silent allow.

<deferred>
## Deferred to v3.4+

- **Full identity provider integration** (JWT/OAuth2/HMAC token signing) — current `X-Session-Scope` header is a lightweight internal scope token. Production-grade auth needs key management, token expiry, and integration with EAASP's identity layer.
- **Multi-scope tokens** (user has multiple scopes, endpoint requires one) — current model is single-scope-per-header.
- **L2 HTTP client connection pooling + retries** — D9 uses simple `httpx.AsyncClient` per-request. Pooling + exponential backoff deferred.
- **D10/D16/D19/D20/D21/D25** — L3 P3 deferred rows (eval frontmatter, file_upload, eval_manager split, serving separation, PolicyEngine contamination, pydantic v2 `globs` migration).
- **D31** — L4 loguru initialization (same D23 pattern, separate phase).
- **OpenTelemetry / structured tracing** — loguru only provides stderr logging. Distributed tracing deferred.
</deferred>
