## Key technical decisions made during execution

- **`s6_subscribe_logs_receives_late_entry` originally wrote a single
  `rx.recv()`** which is racy — the broadcast receiver got "hello"
  (the first emitted line after the warmup), not "late message".
  Decision: deterministic loop with 5-second deadline that iterates
  past early entries until the target arrives. Documented inline.

- **`has_log_buffer` was originally `#[cfg(test)]`** but `show_logs`
  needs it to materialise the per-server buffer before
  `subscribe_logs` (which panics if no buffer exists). Decision:
  removed the test gate; made the method unconditionally public with
  a docstring that mentions the test-only-call-site usage.

- **S6 originally called `StdioMcpClient::connect()` against `sh -c`**
  but `sh` does not speak the MCP initialize handshake required by
  rmcp's `RunningService::serve(transport)`, so the connect always
  fails with "connection closed: initialize response". Decision: keep
  the test exercising the same code path (BufReader → LogEntry →
  push_log_entry → broadcast) by spawning the child via
  `tokio::process::Command::Stdio::piped()` directly. The T3 reader
  task in `stdio.rs` is identical regardless of how the child was
  spawned (and a 5th test, `s6_stdio_client_has_log_manager_builder`,
  exercises the `with_log_manager` API end-to-end via the `connect`
  call site).
