### D-10 — McpManager logs API shape (locked 2026-07-20)

**Pull + push dual API** on `McpManager` (lives at `crates/grid-engine/src/mcp/manager.rs`):

```rust
// Pull — static tail-N
pub async fn take_recent_logs(&self, name: &str, lines: usize)
    -> Result<Vec<LogEntry>>;

// Push — broadcast for live tail
pub fn subscribe_logs(&self, name: &str)
    -> tokio::sync::broadcast::Receiver<LogEntry>;
```

Rationale: `--lines N` is a one-shot read (pull). `--follow` needs ongoing delivery (push via `tokio::sync::broadcast`). `broadcast::Receiver` allows multiple subscribers (e.g. future: `--follow --grep` filter as separate subscriber).

`LogEntry` struct (lives in `crates/grid-engine/src/mcp/log_entry.rs` — new file):

```rust
#[derive(Debug, Clone, Serialize)]
pub struct LogEntry {
    pub timestamp: chrono::DateTime<chrono::Utc>,
    pub level: LogLevel,        // Info | Warn | Error (Debug = Info)
    pub message: String,
}

#[derive(Debug, Clone, Copy, Serialize, PartialEq, Eq)]
pub enum LogLevel { Info, Warn, Error }
```

### D-11 — Buffer strategy (locked 2026-07-20)

**Bounded `VecDeque<LogEntry>` per server**, cap = **1000 entries/server** with drop-old. Backed by `Arc<Mutex<HashMap<String, VecDeque<LogEntry>>>>` in `McpManager`. Plus per-server `broadcast::Sender<LogEntry>` (capacity 256, oldest dropped on overflow).

Rationale: 1000 lines × ~256 bytes/line = ~256 KB/server cap. Predictable memory. Matches `tail -1000` industry default.

### D-12 — `StdioMcpClient` stderr capture (locked 2026-07-20)

Replace `.stderr(std::process::Stdio::null())` at `crates/grid-engine/src/mcp/stdio.rs:65` with `.stderr(std::process::Stdio::piped())`. Bind the resulting `Option<ChildStderr>`, then spawn a tokio task that reads lines (via `tokio::io::BufReader::new(stderr).lines()`) and pushes each parsed `LogEntry` into the `McpManager` buffer + broadcast.

`LogEntry` parser: `level` inferred from stderr line prefix:
- Starts with `ERROR` / `[error]` / `ERR ` → `LogLevel::Error`
- Starts with `WARN` / `[warn]` → `LogLevel::Warn`
- Otherwise → `LogLevel::Info`

This is a best-effort heuristic; documented in API docstring. Exact log format is MCP-server-specific.

### D-13 — CLI UX additions (locked 2026-07-20)

Extend `McpCommands::Logs` variant at `crates/grid-cli/src/commands/types.rs:202-210` with three new fields:

```rust
/// Follow log output (live stream, Ctrl-C to exit)
#[arg(short = 'f', long)]
follow: bool,

/// Filter by log level (info|warn|error)
#[arg(long, value_parser = ["info", "warn", "error"])]
level: Option<String>,

/// Output format (text|json). Honors GRID_OUTPUT_FORMAT env.
#[arg(long, value_parser = ["text", "json"])]
output: Option<String>,
```

Existing `--lines N` retained. `name` stays positional.

`--follow` implementation: spawn a tokio task that calls `subscribe_logs(name)`, receives each `LogEntry`, applies level filter (D-13), formats as text or JSON line, writes to stdout, flushes per line. Ctrl-C handler (already wired in main.rs via `tokio::signal::ctrl_c`) breaks the loop and exits cleanly.

### D-14 — Output format inheritance (locked 2026-07-20)

Default = text (matches Phase 3.7.1 D-05). JSON-line when `--output json` OR when stdout is not a TTY (matches D-07 TTY-aware fallback).

Text format: `2026-07-20T10:23:45Z [INFO ] Server started` (no JSON wrapping).
JSON format: `{"timestamp":"2026-07-20T10:23:45Z","level":"info","message":"Server started"}` (one JSON object per line, matching REQ-AUDIT-05 `to_json()` style).

## Specific Ideas

- The `StdioMcpClient::connect` already binds `_stderr` (underscore-prefixed at line 55). Removing the underscore is the literal one-line change to start capturing.
- `tokio::sync::broadcast::channel(256)` is the idiomatic pattern; Tokio docs show it supports `Sender::send()` returning `Err(SendError)` when no receivers — handle by ignoring (entry still goes to buffer).
- Existing `McpManager::set_runtime_state` pattern (line 132) is the model for thread-safe state mutation under RwLock.
- Plan 03 inherits Plan 02's hermetic test pattern — fake server runs without external deps.

- LogEntry source field (server name) — unnecessary since `subscribe_logs(name)` is already per-server. Defer unless multi-server fan-out is requested.
