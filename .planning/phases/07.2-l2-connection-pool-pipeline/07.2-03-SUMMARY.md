# Plan 07.2-03 Summary: scope filter + archive FTS isolation

## Objective achieved

D11 (skill-registry `scope` filtering applied BEFORE `LIMIT N` instead of as a post-query in-memory filter) and D13 (L2 `archive()` removes the memory from FTS5 search so archived content stops surfacing) — both landed cleanly with passing targeted tests. Phase 7.2 Plan 03 closes the 2 P3 correctness items in the L2 keystone scope.

## What was built / changed

### T01 — D11 access_scope column + migration + backfill (commit `ee4d401a`)

- Added `access_scope TEXT` column to the `skills` table via idempotent `ALTER TABLE` migration (uses `PRAGMA table_info` probe per existing skill-registry migration conventions).
- Created `CREATE INDEX IF NOT EXISTS idx_skills_access_scope ON skills(access_scope)`.
- Added a backfill UPDATE pass: for each existing row, parsed the v2 frontmatter from the skill content and wrote `parsed.access_scope` into the new column. Rows without v2 frontmatter remain `NULL` (semantically equivalent to "no scope filter applies").
- Migration ordering follows the existing skill-registry pattern.

### T02 — D11 scope filter rewrite as SQL WHERE (commit `382d7864`)

- Rewrote `tools/eaasp-skill-registry/src/store.rs::search()` so the scope filter is applied as `AND access_scope = ?` (or `IS NULL` when caller passes `scope=None`) inside the main SELECT's WHERE clause, BEFORE the `LIMIT N`.
- Deleted the post-query in-memory filter loop that was previously at `store.rs:274-290` (per the prior plan-time anchor; actual lines may have drifted after T01).
- Added a scope-filter regression test asserting `scope=X, limit=5` returns exactly the top-5 scope=X matches (no more wasted slots due to post-query filtering of mismatched rows).
- `cargo test -p eaasp-skill-registry` PASS (including the new test).

### T03 — D13 archive FTS isolation (commit `079932cd`)

Per CONTEXT D-05 Option A:

- `MemoryFileStore.archive()` at `files.py:284` now DELETEs every `memory_fts` row for the archived memory_id after the normal `write()` creates the new archived-version row.
- Wrapped in `BEGIN IMMEDIATE` + `get_write_lock` per L2 Phase 7.2 Plan 01 connection-pool conventions.
- Defense-in-depth filter added at `index.py:~270` (FTS path) and `~L375` (HNSW union fetch): `AND mf.status != 'archived'`.
- L2's FTS5 schema at `db.py:102` confirmed non-contentless (standard external-content virtual table), so `DELETE FROM memory_fts WHERE memory_id = ?` is the correct cleanup primitive.

Test `tools/eaasp-l2-memory-engine/tests/test_archive_isolation.py` — 3/3 PASS in 0.16s:

- `test_archived_memory_excluded_from_search` — post-archive search no longer surfaces the memory
- `test_archived_memory_excluded_after_restart` — fresh `HybridIndex` instance after restart still hides the archived memory (persistence held)
- `test_non_archived_memories_still_searchable_after_archive_neighbor` — archiving memory A doesn't affect unrelated memory B's searchability

### Pyright follow-up (commit `b66790e2`)
