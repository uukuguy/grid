### D-02 — D91 HNSW tombstone rebuild: 30% sync threshold
**Decision:** Trigger sync rebuild when `deleted_label_count / total_label_count ≥ 0.30`. Implementation:

1. Track deleted-label count in `HNSWVectorIndex` (likely as `self._deleted_count: int` incremented on each `mark_deleted` call at `vector_index.py:228/247`).
2. On every `add()` call (write path), after the cap-check at `vector_index.py:215`, check `self._deleted_count / self._element_count >= 0.30` (or equivalent denominator). If True, rebuild.
3. Rebuild = create a new `hnswlib.Index` of the same params, replay all live labels (`label not in deleted_set`) via `add_items`, swap pointer, persist via existing `save()` path. Reset `self._deleted_count = 0`.
4. Add a `tools/eaasp-l2-memory-engine/tests/test_tombstone_rebuild.py` pytest covering: (a) under-threshold no-rebuild, (b) exactly-at-threshold triggers rebuild, (c) recall before/after rebuild matches on a small fixed query set (measured).
5. LEDGER close D91 L221 with measurement evidence + commit hash.

**Why sync over async:** Async background rebuild adds task lifecycle (cancel on shutdown / error handling / state during rebuild) for marginal latency gain. Current L2 workload doesn't have proven async-rebuild justification; deferred as P3 if usage pattern emerges. ROADMAP success criterion #2 also implies sync.

**Why 30% threshold:** Matches ROADMAP §"Phase 7.2" success criterion #2 wording verbatim. Common HNSW practice — well below the point where index bloat hurts perf.

### D-03 — D93 embed_batch concurrency: asyncio.gather + per-provider semaphore
**Decision:** Replace the sequential `for text in texts: await embed(text)` pattern in MockEmbedding (provider.py:81), Ollama (provider.py:119), and the third provider (TEI/equivalent, scout will confirm) with:

```python
async def embed_batch(self, texts: list[str]) -> list[list[float]]:
    sem = asyncio.Semaphore(self._concurrency_limit)  # provider-specific
    async def _one(t: str) -> list[float]:
        async with sem:
            return await self.embed(t)
    return await asyncio.gather(*(_one(t) for t in texts))
```

Provider-specific defaults (in code, env-overrideable):
- `MockEmbedding._concurrency_limit = 100` (no real I/O; effectively no limit)
- `OllamaEmbedding._concurrency_limit = 5` (typical Ollama-local default)
- TEI / other: 8 (per typical TEI deployment guidance)

Env override: `L2_EMBED_BATCH_CONCURRENCY` (overrides all providers globally for testing/tuning).

Acceptance per ROADMAP success criterion #3:
- `pytest -k embed_batch` PASS
- batch=10 cost ≤ 30% of sequential (i.e., true concurrency evidence — wall-clock ratio measured)
- Rate-limit-respect test verifies semaphore bound is honored (concurrent in-flight count never exceeds limit)

**Why not native HTTP batch endpoints (Ollama / TEI):** Deferred. Native batch endpoints have richer semantics (error handling per-item, partial batch failures) that warrant their own design pass. `asyncio.gather` gets us ≥3× speedup with minimal LOC; native batch can come in a follow-up perf phase.

### D-04 — D11 skill-registry scope filter: single WHERE clause + new column
**Decision:** Rewrite `tools/eaasp-skill-registry/src/store.rs:213-290` `search()` method to use a single WHERE clause with a normalized `access_scope` column:

1. **Schema migration**: add `access_scope TEXT` column to `skills` table (nullable; backfill from existing rows by parsing v2 frontmatter at migration time).
2. **Index**: add `idx_skills_access_scope ON skills(access_scope)`.
3. **Write path**: when registering/updating a skill, write `parsed_v2.access_scope` into the new column.
4. **Search path**: scope filter becomes `AND access_scope = ?1` (or equivalent) in the main SELECT's WHERE clause; remove the post-query in-memory filter (`store.rs:274-290`). `LIMIT N` now returns accurate scope-filtered top-N.
5. Migration handled via SQLite's standard `ALTER TABLE ADD COLUMN` + a 1-pass UPDATE during the next migration run.

Acceptance per ROADMAP success criterion #4:
- `cargo test -p eaasp-skill-registry` PASS
- Pytest (or Rust test) covers "`scope=X` + `limit=5` returns ≤5 results, all matching scope=X" — assertion PASS.

**Why not post-query keep + multiplier:** Wasteful (over-fetches), doesn't satisfy ROADMAP wording "`LIMIT N` 之前 filter", introduces fragile scaling factor. Adding a column is mechanical and correct.

**Why not two-pass:** Comparable to the WHERE-clause approach but with extra SQL roundtrip. The single WHERE clause is the cleanest path.
