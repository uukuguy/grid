### F2: embed_batch concurrency test tolerance is generous [WARNING]

**Where:** `tools/eaasp-l2-memory-engine/tests/test_embed_batch_concurrency.py:25-32`
**Dimension:** test-quality

**Finding:**
The `test_batch_ten_under_thirty_percent_of_sequential` test asserts `batch_time <= sequential_time * 0.30` (i.e., batch of 10 must be ≤30% of sequential 10). The actual measurement is ~10.09% for mock, well within the bound. However, the 30% bound is a generous ceiling — if the concurrency path silently degrades to sequential (e.g., due to a bug that makes `asyncio.gather()` fan-out but all coroutines run on a single-thread event loop without yielding), the wall time would be ~100% of sequential, which STILL fails the 30% bound. So the test IS sensitive to total concurrency loss. The concern is about *partial* concurrency degradation (e.g., semaphore mis-set to 2 instead of 5): a factor-of-2.5 slowdown would put batch_time at ~25%, which would pass the 30% test silently. 

Real concurrency evidence is confirmed by the `test_semaphore_bound_honored` test (asserts exactly N concurrent coroutines enter via a counter probe — structured lock-step, inherently hermetic). The dual-test design means partial-degradation would be caught by the semaphore-bound test. Acceptable as-is.

**Suggested fix:**
No code change needed. For future phases, consider tightening the tolerance to 20% or adding a parallel-evidence assertion (e.g., "at least 3 coroutines were in-flight simultaneously") to complement the wall-time ceiling.
