## 4. Credibility Gap Analysis

### 唯一阻断可信度的缺口

**外部写入当前不能在副作用前被分类、暂停、批准/拒绝、审计并通过 L4 可观察地呈现。**

完整断链如下：skill frontmatter 没有风险等级 → L3 hook 没有风险等级 → policy engine 没有 gate decision → audit 只有事后 telemetry → L4 没有 request/decision event → CLI 没有同步批准 UX → mock SCADA 只有 read 与永远拒绝的假写。因此“prompt 说不要写”不构成治理。

D-02 固定的正确时序为：

```text
LLM tool call
  → resolve tool + effective hook/risk metadata
  → L3 evaluate_gate（此时尚未 dispatch）
  → allow/approve 才 dispatch actual tool side effect
  → deny 则不调用 tool
  → 追加 decision event/audit
```

门不能提前到 tool 尚未解析时，否则拿不到确定 `tool_name`、风险和 action preview；也不能放到 PostToolUse，否则外部副作用已经发生。

### S8 唯一验证目标

本 phase 只验证 **S8 — Risk-classified external write**：`scada_set_setpoint` 被标为 `write_external`，L3 暂停，CLI 批准，工具随后改变 deterministic fixture 状态，并能观察 request/decision 与 audit。拒绝路径仅是自动测试分支，不是第二个 walkthrough scenario。

### 锁定的非解决方案

以下均禁止进入 Plan 02：

1. 修改 L1 protocol 或让 L1 感知 L3；
2. 新增 L3 HTTP approval endpoint；
3. 实现完整 OPA/Verifier production backend；
4. 实现 multi-party approval；
5. 实现 sandbox-tier enforcement；
6. 实现 L5 Cowork/Web/IM consumer；
7. 增加多个 demo scenarios；
8. 只靠 prompt 声明策略，而不在 pre-side-effect hook 强制。

## 5. Proposed Gate API

### 5.1 Python 类型与精确方法签名

```python
from typing import Literal

from pydantic import BaseModel

RiskLevel = Literal["read", "write_local", "write_external"]
GateDecisionValue = Literal["allow", "approve", "deny", "gate_request"]


class GateDecision(BaseModel):
    decision_id: str
    decision: GateDecisionValue
    rationale: str


async def evaluate_gate(
    self,
    session_id: str,
    hook_id: str,
    tool_name: str,
    risk_level: str,
    action_preview: str,
) -> GateDecision:
    ...
```

单行规范签名（供实现与自动检查引用）：

```python
async def evaluate_gate(self, session_id: str, hook_id: str, tool_name: str, risk_level: str, action_preview: str) -> GateDecision
```

输入边界必须在任何 DB write 前完成：`session_id`、`hook_id`、`tool_name`、`action_preview` 均为非空字符串；`risk_level` 不属于三种闭集时抛出 `ValueError`。不得先创建 audit row 再报参数错误。

### 5.2 Risk × effective mode 决策表

“effective mode”指 latest policy 中相应 hook 的 `mode`，再应用 `managed_hooks_mode_overrides` 的结果。若无风险元数据，按 D-07 解释为 `read`，保持 auto-allow。

| RiskLevel | Effective mode | `evaluate_gate()` 结果 | Audit rationale / 后续 |
|---|---|---|---|
| `read` | `enforce` 或 `shadow` | `allow` | read 始终自动允许 |
| `write_local` | `shadow` | `allow` | 精确 rationale：`shadow mode` |
| `write_external` | `shadow` | `allow` | 精确 rationale：`shadow mode` |
| `write_local` | `enforce` | `gate_request` | CLI 必须解析为 final `approve` 或 `deny` |
| `write_external` | `enforce` | `gate_request` | CLI 必须解析为 final `approve` 或 `deny` |

`approve` 和 `deny` 是 CLI 对 `gate_request` 的最终解析结果，不由初始风险分类直接产生。Deny always wins；`deny` 后不得 dispatch tool。

### 5.3 ManagedHook 字段

在 `ManagedHook` 中加入精确字段：

```python
risk_level: RiskLevel = "read"
```

默认 `read` 保持存量 managed settings 行为。L3 policy 中显式的 hook risk 可覆盖 skill manifest 风险；mode override 仍只改变 mode，不改变 risk taxonomy。

### 5.4 Rust skill manifest 字段

`V2Frontmatter` 使用 Rust serde，而非 Pydantic。精确新增形状为：

```rust
#[derive(Debug, Clone, Default, Serialize, Deserialize, PartialEq)]
#[serde(rename_all = "snake_case")]
pub enum RiskLevel {
    #[default]
    Read,
    WriteLocal,
    WriteExternal,
}

// inside V2Frontmatter
#[serde(default)]
pub risk_level: RiskLevel,
```

若项目 Rust baseline 不允许 enum variant 上 `#[default]`，实现可手写 `impl Default for RiskLevel { fn default() -> Self { Self::Read } }`；wire contract 仍必须是 `read`、`write_local`、`write_external`，且 Default 返回 `Read`。

## 6. Persistence Contract

### 6.1 精确 DDL

```sql
CREATE TABLE IF NOT EXISTS governance_decisions (
    decision_id TEXT PRIMARY KEY,
    session_id TEXT NOT NULL,
    hook_id TEXT NOT NULL,
    tool_name TEXT NOT NULL,
    risk_level TEXT NOT NULL CHECK(risk_level IN ('read','write_local','write_external')),
    decision TEXT NOT NULL CHECK(decision IN ('allow','approve','deny','gate_request')),
    approver TEXT,
    rationale TEXT NOT NULL,
    ts TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE INDEX IF NOT EXISTS idx_governance_decisions_session_ts
    ON governance_decisions(session_id, ts DESC);
```

九列固定为：`decision_id`、`session_id`、`hook_id`、`tool_name`、`risk_level`、`decision`、`approver`、`rationale`、`ts`。Risk 与 decision 各有闭集 CHECK constraint。

### 6.2 AuditStore API 与 transaction 语义

```python
async def record_governance_decision(
    self,
    decision_id: str,
    session_id: str,
    hook_id: str,
    tool_name: str,
    risk_level: str,
    decision: str,
    approver: str | None,
    rationale: str,
) -> GovernanceDecisionOut:
    ...
```

单行规范签名：

```python
AuditStore.record_governance_decision(self, decision_id: str, session_id: str, hook_id: str, tool_name: str, risk_level: str, decision: str, approver: str | None, rationale: str) -> GovernanceDecisionOut
```

实现沿用 telemetry 写入模式：打开连接 → `BEGIN IMMEDIATE` → INSERT → SELECT persisted row → commit-on-success；任意异常 rollback-on-error；最终关闭连接并返回实际持久化的 `GovernanceDecisionOut`。

### 6.3 Append-only 请求与最终决定

表是审计 ledger，不得用 final decision 覆盖 request row。固定采用 append-only ID 规则：

- 请求：`gd_<uuid>`；decision=`gate_request`；
- 最终：`gd_<uuid>_final`；decision=`approve` 或 `deny`；
- final `rationale` 必须链接 request ID，例如 `resolved request gd_<uuid>: interactive CLI`。

这样 `gate_request` 和 final decision 使用不同 primary key，保留完整时序。`allow` 也是独立行，但不需要 final companion row。

## 7. Event and CLI Contracts

### 7.1 L4 governance events

精确调用契约：

```python
await event_stream.append(
    session_id,
    "governance.request",
    {
        "decision_id": decision_id,
        "hook_id": hook_id,
        "tool_name": tool_name,
        "risk_level": risk_level,
        "action_preview": action_preview,
    },
)

await event_stream.append(
    session_id,
    "governance.decision",
    {
        "decision_id": final_decision_id,
        "decision": decision,
        "approver": approver,
    },
)
```

等价的冻结表达：

- `SessionEventStream.append(session_id, "governance.request", {decision_id, hook_id, tool_name, risk_level, action_preview})`
- `SessionEventStream.append(session_id, "governance.decision", {decision_id, decision, approver})`

事件 append 是 **best-effort**：捕获并用 loguru 记录失败；事件失败不得改变已经计算出的 gate result，也不得把已批准动作反转为拒绝。审计 DB 写失败仍按调用方现有错误语义处理，不能伪造成功。

### 7.2 `eaasp session run` CLI 参数

在 primary `run` flow 追加精确 Typer 参数：

```python
yes: bool = typer.Option(
    False,
    "--yes",
    help="Approve governance gates non-interactively",
)
no: bool = typer.Option(
    False,
    "--no",
    help="Deny governance gates non-interactively",
)
```

保持主入口参数流为 `run(message, skill, runtime, no_stream, yes, no)`，不建立旁路命令，也不新增 L3 HTTP endpoint。

规则：

1. `--yes` 与 `--no` 同时出现：参数错误，exit code **2**，不能创建 session。
2. `--yes`：将每个 `gate_request` 解析为 `approve`，approver=`cli:--yes`。
3. `--no`：解析为 `deny`，approver=`cli:--no`。
4. 两者均无：调用 `typer.confirm("Approve?", default=False)`；标准显示包含 **`Approve? [y/N]`**。
5. 交互 yes/no 的 approver 均为 `cli:interactive`。
6. deny：不得调用实际工具，并以非零 exit code **4** 返回。
7. approve：先持久化/发出 final decision，再 dispatch 已解析 tool；副作用不能越过未决 gate。

## 8. S8 Scenario

本计划只定义一个 walkthrough：**S8 — Risk-classified external write**。

### 8.1 Fixture 与 tool contract

```python
def scada_set_setpoint(
    device_id: str,
    setpoint_name: str,
    value: float,
) -> dict[str, object]:
    ...
```

固定输入：

```text
device_id="xfmr-042"
setpoint_name="temperature_limit_c"
value=70.0
risk_level="write_external"
```

S8 skill 必须可部署，其 manifest 把写工具标为 `write_external`。Mock SCADA 以进程内 deterministic state 保存 setpoint，并在成功后返回可断言的新值；不使用真实 endpoint、凭证或客户数据。

### 8.2 唯一成功 walkthrough

```text
agent resolves scada_set_setpoint(xfmr-042, temperature_limit_c, 70.0)
  → resolve skill risk write_external + effective L3 mode enforce
  → evaluate_gate BEFORE tool call
  → append governance.request + gate_request audit row
  → CLI displays action preview and Approve? [y/N]
  → user approves
  → append governance.decision + approve final audit row
  → only now call scada_set_setpoint
  → mock state deterministically becomes 70.0
  → SSE/list-events exposes request and decision
```

walkthrough 成功证据必须同时包括：副作用前 gate、CLI approval、state 从旧值变为 `70.0`、`governance.request`、`governance.decision`、request/final 两条 audit rows。

### 8.3 Denial test（不是第二 scenario）

自动测试还需走同一 S8 的 denial branch：`--no` 或交互 No → final deny event/audit → mock state 不变 → tool call count 为 0 → exit 4。它只是一条 hermetic test branch，不新增 S9 或第二 walkthrough。
