## 2. Current L3 Surface

### 2.1 Managed settings

- `tools/eaasp-l3-governance/src/eaasp_l3_governance/managed_settings.py:L22-L36` 定义 `HookMode = Literal["enforce", "shadow"]` 与 `ManagedHook`。
- `ManagedHook.mode` 当前默认 **`enforce`**（同文件 `:L31-L36`）。
- 当前字段只有 `hook_id`、`phase`、`mode`、agent/skill 选择器和 handler，**没有 `risk_level`**。
- 模型允许额外字段，以保持 richer payload 往返兼容（同文件 `:L26-L36`）。

### 2.2 Policy engine

`tools/eaasp-l3-governance/src/eaasp_l3_governance/policy_engine.py:L65-L70` 定义 `PolicyEngine`；实际导出能力为：

- `deploy()`：append policy version（`:L69-L108`）；
- `switch_mode()`：校验 hook 后 upsert enforce/shadow override（`:L110-L157`）；
- `get_mode_override()`（`:L159-L176`）；
- `list_versions()`（`:L178-L205`）；
- `latest_version()`（`:L207-L230`）；
- `get_version()`（`:L232-L254`）。

当前 **没有 `evaluate_gate()`**，所以风险元数据即使存在也没有确定性决策点。

### 2.3 Audit and database

- `tools/eaasp-l3-governance/src/eaasp_l3_governance/audit.py:L20-L48` 只有 telemetry 输入/输出模型。
- `AuditStore.ingest()` 使用 `BEGIN IMMEDIATE`、成功 commit、异常 rollback（同文件 `:L50-L106`）。
- `AuditStore` 当前持久化的是 PostToolUse telemetry；`query/get/skill_usage` 仍围绕 `telemetry_events`（`:L108-L215`）。**不存在 governance decision ledger。**
- `tools/eaasp-l3-governance/src/eaasp_l3_governance/db.py:L13-L55` 的 schema 只有 managed settings versions、mode overrides 和 telemetry events。
- 每个连接重启 foreign keys，写入由调用处承担 `BEGIN IMMEDIATE`（同文件 `:L58-L83`）。

### 2.4 MCP and REST surfaces

- `tools/eaasp-l3-governance/src/eaasp_l3_governance/mcp_server.py:L19-L93` 精确注册四个 MCP tools：`deploy_managed_hooks`、`list_policy_versions`、`switch_hook_mode`、`validate_session`。
- `_dispatch` 只路由上述四项（同文件 `:L158-L221`）；没有 approval tool。
- `tools/eaasp-l3-governance/src/eaasp_l3_governance/api.py:L4-L11` 列出 health、policy deploy/list/mode、telemetry ingest/query 和 session validate REST surface。
- 实际 route 实现位于 `api.py:L127-L218`；**没有 approval HTTP endpoint**。D-06 要求 3.7.3 继续保持这一事实。

## 4. Credibility Gap Analysis

### 唯一阻断可信度的缺口

**外部写入当前不能在副作用前被分类、暂停、批准/拒绝、审计并通过 L4 可观察地呈现。**

完整断链如下：skill frontmatter 没有风险等级 → L3 hook 没有风险等级 → policy engine 没有 gate decision → audit 只有事后 telemetry → L4 没有 request/decision event → CLI 没有同步批准 UX → mock SCADA 只有 read 与永远拒绝的假写。因此“prompt 说不要写”不构成治理。

D-02 固定的正确时序为：

```text
LLM tool call
  → resolve tool + effective hook/risk metadata
  → L3 evaluate_gate（此时尚未 dispatch）
  → allow/approve 才 dispatch actual tool side effect
  → deny 则不调用 tool
  → 追加 decision event/audit
```

门不能提前到 tool 尚未解析时，否则拿不到确定 `tool_name`、风险和 action preview；也不能放到 PostToolUse，否则外部副作用已经发生。

### S8 唯一验证目标

本 phase 只验证 **S8 — Risk-classified external write**：`scada_set_setpoint` 被标为 `write_external`，L3 暂停，CLI 批准，工具随后改变 deterministic fixture 状态，并能观察 request/decision 与 audit。拒绝路径仅是自动测试分支，不是第二个 walkthrough scenario。

### 锁定的非解决方案

以下均禁止进入 Plan 02：

1. 修改 L1 protocol 或让 L1 感知 L3；
2. 新增 L3 HTTP approval endpoint；
3. 实现完整 OPA/Verifier production backend；
4. 实现 multi-party approval；
5. 实现 sandbox-tier enforcement；
6. 实现 L5 Cowork/Web/IM consumer；
7. 增加多个 demo scenarios；
8. 只靠 prompt 声明策略，而不在 pre-side-effect hook 强制。
