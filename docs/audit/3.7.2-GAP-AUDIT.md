### E-01 — `SessionBar` is mounted only inside the chat branch; no Stop/Resume

**File:** `web/src/components/SessionBar.tsx:22-182` (entire 161-line component)
**File:** `web/src/components/layout/AppLayout.tsx:16` — `{activeTab === "chat" && <SessionBar />}` — chat-only branch
**File:** `web/src/components/SessionBar.tsx:140-181` — root `<div>` renders session pills + `+` button + `ConnectionStatus`. **NO Stop or Resume JSX.**
**File:** `web/src/components/SessionBar.tsx:92-135` — `handleClose` calls `DELETE /api/v1/sessions/:id/stop`. This is session-deletion, NOT stop-running-agent: the `deleteSession` action removes the session from the UI without any graceful cancellation of in-flight LLM calls.

**Reasoning:** D-02 requires Stop/Resume as a *global* controller. The current SessionBar has neither feature. Even Chat-only users cannot stop a runaway agent without killing the entire session.

**UI-SPEC reference:** §9.1 specifies Stop (`Square` icon, `bg-secondary` default with destructive hover), Resume (`Play`, `bg-primary`), live indicator (`bg-primary animate-pulse`).

### E-02 — `ws/types.ts` has no `seq` field

**File:** `web/src/ws/types.ts:7-68`
**Snippet:**
```ts
export type ServerMessage =
  | { type: "session_created"; session_id: string }
  | { type: "text_delta"; session_id: string; text: string }
  // ...
  | { type: "memory_flushed"; session_id: string; facts_count: number }
  // ...
  | { type: "typing"; session_id: string; state: boolean };
```
None of the 13 variants carry a `seq` number. D-04 mandates an optional `seq: u64` field; this is a backward-compatible additive change.

**Downstream consequence:** `web/src/pages/Debug.tsx` cannot display per-message sequence numbers or detect gaps during a reconnection storm. The "observe a running agent" event for Debug is downgraded to MANUAL TUNING because of this gap.

### E-03 — `ws/types.ts` has no `memory_added` event variant

**File:** `web/src/ws/types.ts:7-68`
**Closest existing variant:** `memory_flushed` at L59 with payload `{ session_id: string; facts_count: number }` — this is a periodic summarize event, not a per-write event.

**Downstream consequence:** `web/src/ws/events.ts:24-216` does not have a `case "memory_added":` arm; `web/src/pages/Memory.tsx` cannot wire a live-updating highlight pulse. The Chat "Memory written" toast (D-02 affordance) has no event source.

**Out-of-scope note:** Server-side broadcasting of `memory_added` is owned by the grid-server team (deferred to a separate v3.7.x phase per ADR-V2-024 dual-axis split; the client side is `web/` only per D-01 read-only scope). Plan 02 adds the *client-side* type variant + handler — if the server happens to broadcast `memory_added` already, the wired UI will surface it; if not, the handler is a no-op until that gap is closed separately.

### E-04 — `apps/globals.css` has no `prefers-reduced-motion` rule

**File:** `web/src/globals.css:1-201`
**Snippet (lines 1-22):** `@theme` block declares 16 color tokens. No `@media (prefers-reduced-motion: reduce)` rule exists in the 201-line file (verified by `grep -n 'prefers-reduced-motion' web/src/globals.css` returning 0 matches).

**Downstream consequence:** Users with OS-level reduced-motion preference still see `animate-pulse` (SessionBar live dot, Debug reconnecting state, memory toast potential). UI-SPEC §12.5 mandates this rule. Does NOT block the user-acceptance bar (D-09 ≥ 8.5/10) but is a low-cost fix.
